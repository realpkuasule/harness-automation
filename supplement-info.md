# 我对编程本身的思考

  

从数学形象上来讲，编程其实就是一个收敛的过程。

  

需求的生发是发散的，需求的确认是收敛；

技术调研是发散的（？），技术选型是收敛的；

设计 - 计划 - 任务编排 - 具体的编码 - 测试 - 根据测试结果修正 - 完成任务 …… 都是收敛。

  

因此这个收敛会有很多轮，直到它收敛到一个满足需求、满足设计的状态。

  

# 所以编程不能想到哪就跟ai说一句

  

需要plan，进一步地plan，具体的编排，检查计划，实施，gap analysis，进一步计划，再编排任务，再实施，再gap analysis。

  

# 几个具体有用的原则
* 后端的所有API必须具有同样的数据结构，遵循一致的API设计规范，以便前端拦截器能有效工作。
* 必须使用camelCase字段名，禁止使用snake_case字段名。
* 当发生前端-后端冲突的时候，如果后端满足设计需求，就不要修改后端，而总是修改前端。如果非要修改后端，必须经过用户确认。
* 字体优先选用本地字体，如果考虑到样式的特殊要求，必须使用在先字体的，必须使用中国大陆境内源/镜像，禁止使用境外源。
* 用户测试应该主要是端到端测试，单元测试应该完全由AI自动化完成，集成测试应该尽量由AI完成。
* 关于Docker：必须配置 Docker 国内镜像加速器；把必要的基础镜像保存到本地，不要每次都去网上现拉取。
* 解耦协同：以便不同部分分别开发、互不影响
* 测试先行：先自动生成自动化测试用例auto-bdd
