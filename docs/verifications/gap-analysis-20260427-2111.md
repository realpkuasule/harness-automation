## Minor Issues

### M1. `UNKNOWN_TOOL` error code documented but never returned as structured error

**OpenAPI**: `ErrorCode` includes `UNKNOWN_TOOL` (line 1056)
**Implementation**: `index.ts` line 1220 throws `new Error("Unknown tool: ${name}")` as a generic JavaScript error

When an unknown tool name is received, the MCP SDK catches it and returns an MCP protocol-level error, **not** a structured `HarnessError` payload with code `UNKNOWN_TOOL`. The error code is defined but the handler never constructs a `{ code, message, recoverable }` response for this case.

**Impact**: Low. The MCP SDK delivers the error message to the client. The error code `UNKNOWN_TOOL` is just not reachable through the structured error path.

## Intentional Differences (Documented in OpenAPI)

These are NOT gaps — the OpenAPI spec explicitly documents them as design-vs-implementation differences:

### D1. `techStack` is array (implementation) vs single string (design v3.0)
- OpenAPI line 1169: "设计为单个 string，实现为数组"

### D2. `ProjectPhase` is 4-value enum (vs 3-value in design v3.0)
- OpenAPI line 1025: "实现采用 4 阶段模型以更精确地适配频率评估"

### D3. `TeamSize` is 4-value enum (vs 3-value in design v3.0)
- OpenAPI line 1031: "实现采用 4 级模型以适配频率乘数"

### D4. `RuleMedium` has 3 deprecated aliases beyond design's 8 values
- OpenAPI lines 1005-1008: "claude.md", "linter", "settings.json" marked as backward-compat aliases
- Implementation `normalizeMediumInput()` maps these to canonical values

### D5. `ErrorCode` has 13 values vs design's 10
- OpenAPI line 1058: added `CONFIRM_REQUIRED`, `NO_DECISIONS`, `UNKNOWN_TOOL`

### D6. `GenerateConfigInput` uses `decisions` field (not `recommendations`)
- OpenAPI lines 80, 1184: documented as design/implementation naming difference

### D7. `RollbackInput` uses `backupId` + `list` parameters (not `target`)
- OpenAPI line 318, 1304: documented as design/implementation parameter difference

### D8. AB test tools use flat parameters (not nested `testConfig`)
- OpenAPI lines 677, 993: documented flattening

### D9. `CollectABMetricsInput` uses flat single-point API (not batch with `timeRange`)
- OpenAPI lines 743-744: documented difference
