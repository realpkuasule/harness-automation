## 二、Schema 差异详情

### 🔴 差异 1：DepCheckResult.outdated 类型不匹配 (init_harness 输出)

**严重性**: 中 — 数据格式差异

**OpenAPI 规范** (`openapi.yaml:2310-2315`):
```yaml
outdated:
  type: array
  items:
    type: object
    properties:
      name: string      # 包名
      current: string   # 当前版本
      wanted: string    # 期望版本
```

**实现** (`deps.ts:66-94`):
```typescript
// checkDependencies() 中
let outdated: string[] = [];  // 仅字符串数组
// ...
parsed = JSON.parse(out);    // npm outdated --json 结果
outdated = Object.keys(parsed).slice(0, 10);  // 只提取了包名
```

**影响**: `init_harness` 输出的 `summary.recommendedActions.outdated` 字段值类型为 `string[]`（包名列表），但规范要求 `Array<{name, current, wanted}>`（含版本信息的对象）。下游消费者期望解析 `name`/`current`/`wanted` 属性时会出错。

**根因**: `deps.ts` 的 `checkDependencies()` 定义为 `outdated: string[]`，丢弃了 `npm outdated --json` 中的版本信息。

### 🟡 差异 2：optimize_error_message 对 ruleName 的 fallback 行为

**严重性**: 低 — 边界行为差异

**OpenAPI 规范** (`openapi.yaml:923`):
> "不提供 ruleId 或 scenario 时 suggestions 为空数组"

**实现** (`index.ts:1170-1217` + `error_optimization/generator.ts:37-96`):
```typescript
// generateErrorSuggestion() 有三条匹配路径：
// 1. ruleId → 精确匹配 (confidence: 0.95)
// 2. scenario → 场景匹配 (confidence: 0.85 / 0.6)
// 3. ruleName → 模糊匹配 (confidence: 0.5) ← 规范未提及
if (context.ruleName && !context.ruleId) {
  const nameMatch = getAllTemplates().filter((t) => {
    // ruleName 分词后模糊匹配模板名
  });
}
```

**影响**: 当仅提供 `ruleName`（不提供 `ruleId` 和 `scenario`）时，`suggestions` 可能非空，与规范断言矛盾。实际行为更灵活，但规范文档未反映。

### 🟢 差异 3：validate_setup checkFiles 默认值未声明

**严重性**: 信息性

**OpenAPI 规范**: `checkFiles` 的 schema 中无 `default` 字段，仅在 description 中提及 9 个默认文件。

**实现** (`validators/setup_validator.ts:42-52`):
```typescript
const MANAGED_FILES = [
  "CLAUDE.md", "eslint.config.json", ".claude/settings.json",
  ".gitignore", ".husky/pre-commit", ".husky/commit-msg",
  ".github/workflows/ci.yml", "package.json", ".harness/state.json",
];
// 当 checkFiles 未传入时使用此默认值
```

**建议**: 规范中为 `checkFiles` 添加 `default` 声明以明确默认行为。

### 🟢 差异 4：init_harness dryRun 输出缺少 summary 可选字段

**严重性**: 信息性

**OpenAPI 规范** (`openapi.yaml:247-253`): `InitHarnessSummary` 包含 `backupDir`、`recommendedActions`、`huskySetup` 等字段（均为 nullable）。

**实现**: `dryRun=true` 时返回的 `summary` 仅包含 `files` 和 `decisions`，缺少 `backupDir`/`recommendedActions`/`huskySetup`。这是合理的（dry-run 不做文件操作），但规范未明确说明 dryRun 下哪些 summary 字段缺失。

### 🟢 差异 5：reset_state 描述措辞

**严重性**: 信息性

**规范**: "将状态机的 phase 重置为 null，用于重新开始流程。"
**实现**: "重置 Harness 状态（用于重新开始流程）"

实现省略了 `phase 重置为 null` 的技术细节。功能行为一致。

## 四、行为差异

### 4.1 错误码使用一致性

| 场景 | 规范要求 | 实现使用 | 一致性 |
|------|---------|---------|--------|
| get_rule_stats 无引擎输出 | 400 错误 | `STATE_NOT_FOUND` | ✓ |
| analyze_rule_adjustments 无分析数据 | 400 错误 "缺少分析前置数据" | `DEPENDENCY_MISSING` | ✓（语义匹配） |
| cognitive_skill 规则未找到 | 400 错误 "规则未找到或输入无效" | `INVALID_CONFIG` 或 `UNKNOWN_ERROR` | ✓ |
| export_rules 无引擎输出 | 400 错误 "未找到引擎输出" | `STATE_NOT_FOUND` | ✓ |
| rollback list=true 无备份 | `ROLLBACK_FAILED` | `ROLLBACK_FAILED` | ✓ |

### 4.2 scan_codebase 缓存模式

**规范**: "支持增量扫描缓存（useCache=true）" — 仅在 description 中提及。

**实现**: `scanners/scan_cache.ts` 实现了完整的文件级缓存系统，基于 mtime + size 判断文件变更。`useCache=true` 时调用 `scanDirCached()`，`scanDepth="quick"` 时调用 `scanConfigOnly()`。行为正确，实现超出规范描述范围但兼容。

### 4.3 import_rules 数据丢失

**规范**: `import_rules` 返回 `RuleDecision[]`。

**实现**: `io/rule_io.ts:207-237` 的 `importRules()` 重建的 `RuleDecision` 不包含 `adjustedCost`、`adjustedCostLabel`、`feedbackSpeed`、`errorMessage` 字段（均为 Optional）。这是合理的导出/导入数据截断，不违反 Schema 约束。
