### 2.2 未标注的细微差异

#### a) `generate_config` 的 `decisions` 接受空数组并回退到 state

规约：`decisions` 是 `required` 的数组。
实现（`index.ts:260-262`）：接受 `decisions: []`，此时回退到 `state.engineOutput.decisions`。只有当两者都为空时才返回 `NO_DECISIONS` 错误。该容错行为未在规约中描述。

**影响**: 低。该行为向后兼容，增强了健壮性。

#### b) `generate_config` 和 `confirm_decisions` 通过启发式识别精简版 decisions

实现（`index.ts:638-641`）：检查 `input.decisions[0].confidence === undefined` 来判断是否是仅含 `ruleId` + `recommendedMedium` 的精简版。该判断逻辑未在规约中描述。

**影响**: 低。内部实现细节，不影响外部契约。

## 3. 输出 Schema 差异

### 3.1 ⚠️ `init_harness` dryRun 模式的输出形状与规约不匹配

**规约** (`openapi.yaml:243-251`)：
```yaml
properties:
  files:
    type: array
    items:
      $ref: "#/components/schemas/ConfigFileResult"
  summary:
    $ref: "#/components/schemas/InitHarnessSummary"
  validation:
    $ref: "#/components/schemas/ValidateSetupOutput"
```

**实现**（`index.ts:491-503`）dryRun 路径返回：
```json
{
  "phase": "evaluate",
  "decisions": [...],
  "summary": { "total": N, "byMedium": {...}, "highConfidence": N, "cognitiveRequired": N },
  "message": "Dry run — no config generated. Re-run without --dryRun to generate."
}
```

dryRun 时没有 `files` 和 `validation` 字段，多了 `phase`、`decisions`、`message`。

**严重性**: 中。调用方如果按规约解析 dryRun 输出会预期 `files` 数组。

### 3.2 ⚠️ `init_harness` 非 dryRun 的 `files` 包含 husky/CI 文件

实现动态将 husky hooks 和 CI workflow 文件追加到 `files` 数组（`index.ts:521-539`）。规约的描述中提到生成 "CLAUDE.md、ESLint 配置、.claude/settings.json、.gitignore 四种核心配置文件"，但实际响应中包含额外的 .husky/ 和 .github/ 文件。规约的 `InitHarnessSummary` 没有限制文件类型，且 `ConfigFileResult` 足以描述这些额外文件。

**严重性**: 低。规约响应 schema 可以兼容。

### 3.3 `import_rules` 的非 preset 导入路径输出缺少 `preset`/`presetName`

当使用 `exportJson` 或 `filePath` 导入时，实现返回 `{ decisions, total, warnings }`（`index.ts:988-1015`），无 `preset`/`presetName` 字段。规约输出 schema 将这些字段列为可选（非 `required`），因此是兼容的。

**严重性**: 信息性。兼容。

### 3.4 `cognitive_skill` 缺省处理 — 实现包装 `{error}` 而非直接抛异常

规约的 400 响应描述 `HarnessError` schema。实现（`index.ts:1124-1131`）将 `processCognitiveRequest` 抛出的异常捕获并包装为 `{ error: { code: "UNKNOWN_ERROR", message, recoverable: true } }` 返回。规约没有说明这种情况会返回 200 而非 400（MCP 层面两者都是 JSON-RPC 响应）。

**严重性**: 低。错误信息结构匹配 `HarnessError` schema，但 HTTP 状态码概念在 MCP 中不存在。

## 5. 行为差异

### 5.1 ⚠️ `optimize_error_message` 的 `rateAfter` 参数未被使用

规约（`openapi.yaml:933`）接受 `rateAfter: boolean`。
实现的 Zod schema（`types.ts:490`）也定义了 `rateAfter: z.boolean().optional()`。
但 handler（`index.ts:1159-1183`）始终调用 `evaluator.recordSuggestion()`，未检查 `rateAfter` 参数。

**严重性**: 中。客户端如果设置 `rateAfter: false` 期望不记录评价，但实际仍会记录。

### 5.2 `reset_state` 将 phase 设为 `null` 而非 `"reset"`

规约（`openapi.yaml:147`）："将状态机的 phase 重置为 null"。
实现（`index.ts:414`）：调用 `stateManager.updateStatus(null)`。

✅ 规约与实现一致。但实现的 `HarnessStatus` 类型包含 `null`，没有 `"reset"` 状态，与规约的 `nullable: true` 枚举一致。

### 5.3 `list_rule_presets` 使用空字符串 `projectDir`

实现（`index.ts:1032`）：`const io = new RuleIO("")`。规约没有要求 `projectDir` 参数（输入 schema 仅有可选的 `techStack`）。该实现可行因为 `RuleIO.listPresets()` 不依赖 `projectDir`。

**严重性**: 信息性。可行的内部实现选择。

### 5.4 `query_state` 的条件性字段输出

实现选择性包含字段（如 `project`、`lastEvalAt`、`confirmedAt`、`validatedAt`、`sessionId`、`generationLog`、`validation` 等），仅在对应数据存在时返回。规约的 `QueryStateOutput` schema 将这些字段全部列为可选。

✅ 规约与实现一致。

## 8. 建议

### 高优先级
1. **修复 `init_harness` dryRun 输出形状** — 使 dryRun 输出包含与规约一致的字段（或更新规约以反映实际行为）。
2. **修复 `optimize_error_message` 未检查 `rateAfter`** — 在 handler 中添加条件判断，尊重 `rateAfter: false` 时跳过评价记录。

### 中优先级
3. **补充规约中省略的错误码** — 考虑在 `openapi.yaml` 的各工具 400 响应 description 中列出具体的 `ErrorCode` 值。
4. **更新 `SuitabilityWarning.type` 移除未使用的 `cost`** — 类型定义包含未使用的枚举值，可考虑移除或实现 cost 类型的检查。

### 低优先级
5. **为 `generate_config` 的回退行为添加文档** — 规约应注明 `decisions: []` 会触发 state 回退。
6. **规约中 `init_harness` 输出增加 husky/CI 文件的描述** — 当前描述仅提 4 种核心文件，但实际输出包含额外文件。
