## 发现的问题

### 问题 1: generate_config 与 init_harness 的 enrich 路径不一致

`generate_config` 生成的 CLAUDE.md 中规则描述为 `"User-configured medium assignment"` 占位符，而 `init_harness` 生成的版本有完整描述（如 "规则可形式化，适合自动化检查"）。

**根因**: `generate_config` 中处理用户传入的精简决策时，调用 `enrichPartialDecisions()` 查找 rules.json，但该函数返回的 `reasons` 固定为 `["User-configured medium assignment"]`，未从 rule 定义读取实际描述。而 `init_harness` 直接从 `DecisionEngine.evaluate()` 输出获取完整决策，其 `reasons` 包含正确的评估理由。

### 问题 2: eslint.config.js 为 CommonJS 格式，项目可能为 ESM

生成的 `eslint.config.js` 使用 `module.exports = [...]`（CommonJS），但在 ESM 项目（package.json 含 `"type": "module"`）中 ESLint 10 flat config 需要 `export default [...]`。

### 问题 3: .lintstagedrc.json 中 prettier 命令参数互斥

```json
"*.{json,md,yaml,yml}": ["prettier --write --check"]
```

`--write` 和 `--check` 在 prettier 中是互斥参数。`--write` 执行格式化并写入文件，`--check` 仅检查格式不写入。

### 问题 4: commit-msg hook 缺少 commitlint.config.js

Hook 内容为 `npx --no -- commitlint --edit $1`，但 `init_harness` 未生成 `commitlint.config.js`。虽然 `@commitlint/config-conventional` 已安装，没有配置文件时 commitlint 会因找不到配置而失败。

### 问题 5: settings.local.json 权限白名单不完整

只有 2 个工具被允许，其余 MCP 工具每次调用都需要用户手动确认。

### 问题 6: CI 工作流对 prototype 阶段偏重

prototype/solo 阶段生成了完整的 CI 流水线（`ubuntu-latest, Node 18 matrix, lockfile-lint, npm test, build`），但 solo prototype 项目通常没有 CI runner 触发，属于配置浪费。

## 改进建议优先级

| 优先级 | 问题 | 影响面 |
|--------|------|--------|
| P0 | eslint.config.js 格式兼容性 | 运行时即可暴露，用户会直接感知 |
| P0 | commit-msg hook 缺少配置 | hook 运行即失败 |
| P1 | .lintstagedrc.json prettier 命令错误 | lint-staged 运行失败 |
| P1 | enrich 路径不一致（占位符 vs 实义描述） | 用户可见的文本质量差异 |
| P2 | prototype CI 偏重 | 非功能性，影响不大 |
| P2 | permissions 白名单不完整 | 体验问题，不阻塞功能 |
