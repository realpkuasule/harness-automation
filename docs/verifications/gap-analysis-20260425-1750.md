## 4. 响应 Schema 差异

### 4.1 `start_ab_test` 响应中 `endTime` 的必需性

**位置**: Spec L699-706 / `mcp-server/src/ab_test/manager.ts:15-22`

| 方面 | Spec | 实现 |
|------|------|------|
| `endTime` | 总是存在 (`required`) | 可选 (`?`)，仅过期测试返回 |

**影响**: 低。响应模型应反映 `endTime` 为可选/可空。实际返回的 JSON 中，仅当 `status === "already_running"` 且测试已过期时才会有 `endTime`。刚启动的测试、配置无效的情况均无 `endTime`。

### 4.2 `RollbackOutput.cleaned` 的可选性

**位置**: Spec L1655-1659 / `mcp-server/src/types.ts:378-385`

| 方面 | Spec | 实现 |
|------|------|------|

**影响**: 低。实际上当 `cleaned` 为空数组时，实现使用 `null`（与 spec 一致）；当没有任何文件需要清理时，字段可能完全被省略。Spec 应标记为 `nullable: true` 且不需要 `required`。

### 4.3 `init_harness` 响应缺少文件写入错误

**位置**: Spec L238-252 / `mcp-server/src/index.ts:567-579`

实现中 `init_harness` 在文件写入时收集错误到 `errors` 数组：
```typescript

// ... catch (e) { errors.push(...); }
```

但**这些错误未包含在响应中**（仅通过 `sm.setConfigOutput({ ..., errors, warnings: [] })` 保存到 state）。Spec 中响应 schema 也未定义 `errors` 字段。

| 方面 | Spec | 实现 |
|------|------|------|
| 响应中的 `errors` | 未定义 | 不存在（仅存于 state） |

**影响**: 中。文件写入失败在 `init_harness` 调用者侧不可见。应考虑：
1. 在响应中加入 `errors` 字段，或
2. 在 Spec 中显式声明此行为
