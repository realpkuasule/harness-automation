## 2. Schema Discrepancies

### 2.1 `OptimizeErrorMessageInput.projectDir` — Required Mismatch

| Source | Required? | Detail |
|--------|-----------|--------|
| **Spec** | Yes | `required: [projectDir]` (line 936) |
| **Impl** | No | `projectDir: z.string().optional()` (types.ts:497) |

The implementation accepts calls without `projectDir`; when absent, it skips recording evaluation data (`rateAfter` behavior). The spec unconditionally requires it.

**Impact**: Low. Clients following the spec strictly will always pass `projectDir`; the implementation handles both.

---

### 2.2 `start_ab_test` Response — `endTime` Optionality

| Source | Field | Detail |
|--------|-------|--------|
| **Spec** | `endTime` | Required, `format: date-time` (line 722) |
| **Impl** | `endTime` | `endTime?: string` (manager.ts:21) — optional |

`endTime` is only set when `status === "started"`. For `already_running` / `invalid_config`, it may be omitted or undefined.

**Impact**: Low. Clients should handle missing `endTime`.

---

### 2.3 `init_harness` Output — `errors` Omitted When Empty

| Source | Behavior |
|--------|----------|
| **Spec** | `errors: array of ConfigError` — always present |
| **Impl** | `errors: errors.length > 0 ? errors : undefined` — omitted when empty |

(index.ts:591) The `errors` field is set to `undefined` (stripped by JSON.stringify) when no errors occurred, rather than an empty array `[]`.

**Impact**: Low. Clients expecting an array may get `undefined`.

---

### 2.4 `optimize_error_message` Output — `stats` Conditionally Omitted

| Source | Behavior |

(index.ts:1182-1188) Stats recording and output is conditional on `projectDir` being set and `rateAfter !== false`.

**Impact**: Low. Same pattern as 2.3.

## 3. Behavioral / Semantic Gaps

### 3.1 `evaluate_rules` — Two-State Saves

The spec describes saving to `.harness/state.json` in one step. The implementation calls `stateManager.setEngineInput(input)` then `stateManager.setEngineOutput(output)` separately, each triggering a full `writeFileSync`. On dry-run, no state is written — matching spec. The two-write pattern is an implementation detail with no external effect.

**Impact**: None. Observable behavior matches spec.

---

### 3.2 `generate_config` — `decisions` Fallback Logic

The spec documents the input field `decisions` (vs design's `recommendations`) and says it's required. The implementation additionally falls back to `state.engineOutput.decisions` when `input.decisions` is empty:

```typescript
const decisions = input.decisions.length > 0
  ? enrichPartialDecisions(...)
  : (state.engineOutput?.decisions ?? []);
```

This fallback is not documented in the spec but is functionally transparent — clients always control behavior by passing `decisions`.

**Impact**: Low. Not a violation, but the fallback behavior is undocumented.

---

### 3.3 `query_state` — Conditional Field Emission

The implementation conditionally adds fields to the response (only present if the underlying state has them):

```typescript
result.project = state.project;       // conditional
result.lastEvalAt = state.updatedAt;  // conditional
result.confirmedAt = state.confirmedAt; // conditional
result.validatedAt = state.validatedAt; // conditional
result.validation = state.validation;   // conditional
result.sessionId = state.sessionId;     // conditional
result.generationLog = state.generationLog; // conditional
result.summary = summary;               // always present, possibly empty
```

The spec shows all fields at the same level without `required`, so this is technically compatible with OpenAPI. However, a client expecting, e.g., `sessionId` to always be present may get `undefined`.

**Impact**: Low. Document the conditional nature.

## 4. Internal Type Consistency Notes

These are not spec-impl gaps, but internal inconsistencies worth noting:

### 4.1 `RuleExportData` Omits Enriched Fields

When exporting (`export_rules`), the `RuleExportData` type does not include `errorMessage`, `adjustedCost`, `adjustedCostLabel`, or `feedbackSpeed`. On re-import, these fields are reconstructed from rule definitions rather than preserved from the export. The spec does not define the export schema in detail, so this is not a gap, but it means round-trip fidelity is lossy for computed fields.

### 4.2 `HarnessState.validation.findings` Computed

`state.validation.findings` is computed as `errors + warnings + info`. The spec shows `findings: integer`, consistent with this computation. ✅
