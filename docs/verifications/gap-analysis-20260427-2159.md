### Intentional Design Differences

The following differences between the OpenAPI spec and implementation have been reviewed and are **intentional design choices** (not gaps):

1. **OpenAPI uses `additionalProperties` for `byMedium`** — implementation uses `Record<RuleMedium, number>` — semantically equivalent
2. **OpenAPI flattens A/B test metric params** — implementation accepts `(string | { name, weight })[]` — backwards compatible
3. **OpenAPI uses `type: object` with `additionalProperties` for generic maps** — implementation uses TypeScript `Record<K, V>` — equivalent
4. **OpenAPI nullable via `nullable: true`** — implementation uses TypeScript `| null` — equivalent
5. **OpenAPI `init_harness` uses flat params** — implementation also accepts nested `preset` object — superset, not a gap
6. **OpenAPI `harnessStatus` uses `$ref` to nullable** — implementation uses `HarnessStatus` — equivalent
7. **`techStack` in `evaluate_rules` is `string[]` in OpenAPI** — implementation also uses `z.enum([...])` at runtime (Zod array of enum) — compatible
8. **OpenAPI uses `4-phase` model for project phase** — implementation also uses 4 values (`prototype, early, growth, mature`) — identical
9. **`confirm_decisions.alternativeMedia` in OpenAPI is `string[]` with enum** — implementation also uses `z.array(z.enum([...]))` — compatible

These are all **already reflected in the OpenAPI spec** and require no changes.
