## 3. New Gaps Found

### 3.1 `CognitiveAutoTrigger` Type Boundary (Minor)

| Source | Type Definition |
|--------|----------------|
| **Spec** | `CognitiveAutoTrigger`: constrained 4-field schema — `skillType: "educational"`, `ruleId`, `topic`, `experienceLevel` |
| **Impl** | `shouldAutoTrigger()` returns `CognitiveRequest | null`, where `CognitiveRequest` has 11+ optional fields (`projectDir`, `codePattern`, `currentMedium`, etc.) |

**Runtime behavior**: ✅ Match — `shouldAutoTrigger()` populates only the 4 spec fields, so JSON output is identical.

**Risk**: If `shouldAutoTrigger()` is later modified to include extra fields (e.g., `codePattern`), the response would silently deviate from spec. Consider either:
- Creating a dedicated `CognitiveAutoTrigger` type in the impl
- Adding `CognitiveAutoTrigger` as a documented interface

**Severity**: Low (no current behavioral gap)

### 3.2 `query_state.lastEvalAt` Uses `state.updatedAt` (Minor Behavioral Gap)

| Source | Behavior |
|--------|----------|
| **Spec** | Field `lastEvalAt` — implies dedicated evaluation timestamp |
| **Impl** | `index.ts:375-377` — sets `result.lastEvalAt = state.updatedAt` (generic "last state save" time) |

**Problem**: After `confirm_decisions`, `setProjectInfo`, or any other state mutation, `updatedAt` advances past the evaluation time. If a caller reads `lastEvalAt` after confirmation, they get the confirmation time, not the evaluation time.

**Example sequence**:
1. `evaluate_rules` → `updatedAt = T1` ✅
2. `confirm_decisions` → `updatedAt = T2` (> T1)
3. `query_state` → `lastEvalAt = T2` (≠ evaluation time)

**Severity**: Low. No client currently depends on `lastEvalAt` being the exact eval timestamp. The spec description for `query_state` already notes fields are "仅在对应数据存在时返回，不保证始终存在".

### 3.3 `optimize_error_message.projectDir` Dual Role Undocumented

| Aspect | Documented | Actual |
|--------|-----------|--------|
| Evaluation data recording | ✅ Spec description: "不传时不记录评价数据" | ✅ `index.ts:1182-1188` — records to evaluator |
| Trigger history recording | ❌ Not documented | `index.ts:1192-1204` — writes to `.harness/trigger_history.json` |

The trigger history side effect is a write to `.harness/trigger_history.json` that happens whenever `projectDir` is set (regardless of `rateAfter`). This is a persistent side effect not described in the tool's specification.

**Severity**: Low. The history file is internal to the harness system and doesn't affect visible API contract.

### 3.4 `init_harness` Preset Not Actually Simplifying

| Source | Claim |
|--------|-------|
| **Spec** | `preset` field description: "嵌套 preset 对象（实现新增，支持简化调用）" |
| **Impl** | `InitHarnessInputSchema` requires `projectPhase`, `teamSize`, `techStack` as flat fields; preset merge only happens AFTER Zod validation |

**Problem**: The "简化调用" (simplified calling) claim is misleading. A caller cannot omit the flat params and use only preset — Zod will reject the input. The preset acts as a parallel override, not a simplification.

```
// This FAILS despite preset being specified:
{ projectDir: ".", preset: { techStack: ["typescript"], projectPhase: "early", teamSize: "small" } }
// Zod requires projectPhase/teamSize/techStack at root level
```

**Severity**: Low. Documentation claim vs implementation behavior. No behavioral correctness issue.

### Resolution Recommendations

1. **§3.1 (`CognitiveAutoTrigger` type)**: Add a dedicated `CognitiveAutoTrigger` interface in `orchestrator.ts` and use it as the return type of `shouldAutoTrigger()` instead of the broader `CognitiveRequest`. This prevents accidental field leakage.

2. **§3.2 (`lastEvalAt`)**: Store a dedicated `lastEvalAt` timestamp in `HarnessState` when `setEngineOutput` is called, separate from `updatedAt`.

3. **§3.3 (`projectDir` dual role)**: Update the OpenAPI spec description for `optimize_error_message.projectDir` to mention trigger history recording as a side effect.

4. **§3.4 (preset simplification)**: Either relax Zod validation when `preset` is provided, or update the spec description to accurately describe preset as a merge/override mechanism rather than a simplification.

**None of the 4 gaps affect correctness or test outcomes.** All 320 tests pass with the current implementation.
