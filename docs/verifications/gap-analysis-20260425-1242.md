## 2. 架构缺口详情

### 2.1 冲突检测未实现

- **设计位置**: §6.2 冲突检测
- **设计内容**: 规则冲突矩阵 + `detectConflicts()` 函数
- **当前状态**: `engine.ts` 中无任何冲突检测逻辑
- **影响**: 当用户同时启用冲突规则（如 `no-explicit-any` + `allow-any-in-prototype`）时，引擎无提示

### 2.2 `dryRun` 模式缺失

- **设计位置**: §3.1.3 `GenerateConfigInput.dryRun`、§5.3 Dry-run 模式
- **设计内容**: `dryRun?: boolean`，为 true 时只预览不写文件
- **当前状态**: `generate_config` 无 `dryRun` 参数
- **影响**: 用户无法在写入前预览生成的文件内容

### 2.3 状态机使用 `status` 而非 `phase`

- **设计位置**: §4.1 `HarnessState.phase`、§4.2 状态机
- **设计内容**: 使用 `phase` 字段 + 数值索引 (`null(0) → evaluated(1) → confirmed(2) → generated(3) → validated(4)`)
- **当前状态**: 使用 `status` 字段，无阶段顺序校验
- **影响**: 无法防止状态跳转（如从 `null` 直接到 `confirmed`），无断点续做的阶段判断

### 2.4 错误码体系未实现

- **设计位置**: §5.1 错误码定义
- **设计内容**: 结构化 `ErrorCode` 枚举 + `MCPError` 接口（含 `code`、`message`、`detail`、`recoverable`）
- **当前状态**: 无标准错误码，错误信息为自由文本
- **影响**: 客户端无法根据错误码做自动化恢复

### 2.5 `generate_config` 不写文件

- **设计位置**: §3.1.3 行为描述
- **设计内容**: `generate_config` 负责写文件 + 备份，返回每个文件的 `action` 和 `backupPath`
- **当前状态**: 只返回文件内容列表，写入责任推给了客户端
- **影响**: 与设计核心约定（MCP 层负责文件操作）不一致

### 2.6 SKILL.md 过于简化

- **设计位置**: §9 SKILL.md
- **设计内容**: 完整的 11 步工作流，含 AskUserQuestion 交互、断点续做、错误恢复引导
- **当前状态**: 6 步简化版，缺少用户交互细节和错误恢复引导
- **影响**: 用户体验不如设计预期顺畅

### 2.7 `assess_suitability` 未实现

- **设计位置**: §3.1.9
- **设计内容**: 评估是否适合应用 harness（原型检测、脚本识别、成本评估）
- **当前状态**: 无 `suitability/` 模块
- **影响**: 无法在不适合的场景中自动提供轻量建议（Phase 4 范围，不影响核心流程）

### 2.8 Phase 4 功能未实现（按设计）

- **设计位置**: §10 Phase 4
- **设计内容**: A/B 测试框架、认知层 Skills、错误信息优化
- **当前状态**: 全部未实现
- **影响**: 按设计属于可选扩展，不影响核心流程

## 3. 接口差异明细

### 3.1 `evaluate_rules` 输出对比

| 字段 | 设计 | 当前实现 | 匹配 |
|------|------|---------|------|
| `.recommendations` | 完整 `RuleRecommendation` | `decisions[]` | ❌ 命名不同 |
| `.recommendations[].conflicts` | `RuleConflict[]` | 无 | ❌ 缺失 |
| `.recommendations[].adjustedCost` | `critical/high/medium/low` | 无 | ❌ 缺失 |
| `.recommendations[].feedbackSpeed` | `immediate/commit/pr` | 无 | ❌ 缺失 |
| `.recommendations[].evidence` | 字符串 | 无 | ❌ 缺失 |
| `.recommendations[].errorMessage` | `{why, whatInstead, reference}` | 无 | ❌ 缺失 |
| `.summary.byMedium` | `Record<string, number>` | ✅ | ✅ |

### 3.2 `generate_config` 输出对比

| 字段 | 设计 | 当前实现 | 匹配 |
|------|------|---------|------|
| `.files[].action` | `created/overwritten/skipped/merged/dry_run` | 无此字段 | ❌ 缺失 |
| `.files[].backupPath` | 字符串 | 无 | ❌ 缺失 |
| `.errors` | `ConfigError[]` | 无 | ❌ 缺失 |

### 3.3 `init_harness` 输入对比

| 参数 | 设计 | 当前实现 | 匹配 |
|------|------|---------|------|
| 结构 | `preset: {techStack, projectPhase, teamSize}` | `techStack[] + projectPhase + teamSize` | ⚠️ 扁平 vs 嵌套 |

## 4. 已实现但设计未覆盖的功能

当前实现包含以下不在原设计中的功能：

- `reset_state` — 清除项目状态
- `get_rule_stats` — 规则效果统计（Phase 3）
- `analyze_rule_adjustments` — 自适应调整建议（Phase 3）
- `export_rules` / `import_rules` — 规则导入导出
- `list_rule_presets` / `list_rule_exports` — 预设管理

这些表明实现已经部分超越了 Phase 1-2 的范围，进入了 Phase 3 领域。

## 5. 优先级建议

### P0 — 核心流程缺口

| 缺口 | 工作量估计 | 说明 |
|------|-----------|------|
| 冲突检测 | 小（~0.5天） | 在 `engine.ts` 中添加冲突矩阵 + 检测逻辑 |
| `dryRun` 模式 | 小（~0.5天） | 在 `generate_config` 中添加参数 + 分支 |
| 状态机 phase 规范化 | 中（~1天） | `status` → `phase` + 阶段校验 + 数值索引 |
| 错误码体系 | 中（~1天） | 定义 `ErrorCode` 枚举 + 统一 MCP error 格式化 |

### P1 — 用户体验改进

| 缺口 | 工作量估计 | 说明 |
|------|-----------|------|
| SKILL.md 补全 | 小（~0.5天） | 加入 AskUserQuestion 交互编排 + 断点续做逻辑 |
| `generate_config` 写文件 | 中（~1天） | 对齐设计约定，MCP 层负责文件操作 |

### P2 — Phase 4 可选功能

| 缺口 | 工作量估计 | 说明 |
|------|-----------|------|
| `assess_suitability` | 中（~1天） | 模块开发 + 集成 |
| A/B 测试框架 | 大（~3天） | 3 个工具 + 数据收集 + 统计分析 |
| 认知层 Skills | 大（~3天） | 触发机制 + 上下文传递 + 3 种 Skill 类型 |
| 错误信息优化 | 中（~2天） | 模板库 + 上下文生成 + 效果评估 |
