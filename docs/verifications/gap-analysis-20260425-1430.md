## 2. 输入 Schema 差异 (Input Schema Gaps)

### 2.1 init_harness — `projectPhase` / `teamSize` / `techStack` 必填性不一致

| 来源 | projectPhase | teamSize | techStack |
|------|-------------|----------|-----------|
| **OpenAPI** | optional (仅 projectDir 必填) | optional | optional |
| **实现 (Zod)** | **required** | **required** | **required** |

**差异**: OpenAPI 声明只有 `projectDir` 是 required，但 Zod Schema 要求 `projectPhase`, `teamSize`, `techStack` 全部必填。这意味着 OpenAPI 允许 `init_harness` 只传 `projectDir`（结合 preset 使用），但实际代码会拒绝这种调用。

**Side effect**: `preset` 参数的用途被削弱 — OpenAPI 描述它为"支持简化调用"的嵌套参数，但由于顶层字段已全部必填，通过 `preset` 传参的意义有限。

### 2.2 import_rules — 三选一约束未在 Schema 层强制执行

| 来源 | 约束 |
|------|------|
| **OpenAPI** | "三者至少提供一个"（presetId / exportJson / filePath） |
| **实现 (Zod)** | 三者均为独立 optional，无互斥/至少选一约束 |

**差异**: OpenAPI 文档声明至少要提供三者之一，但 Zod Schema 未做此约束。运行时 handler 中会检查并返回 `INVALID_CONFIG` 错误，类型系统未体现此约束。

### 2.3 start_ab_test — baselineMedium / testMedium 枚举范围

| 来源 | 枚举值 |
|------|--------|
| **OpenAPI** | `$ref: RuleMedium` (11 个值，含 deprecated 别名) |
| **实现** | 明确列出 8 个推荐值 + 3 个废弃别名，但 handler 中会调用 `normalizeMediumInput` 做转换 |

**差异**: 实际不存在明显差异 — 实现接受的枚举值与 OpenAPI 一致。

## 3. 输出 Schema 差异 (Output Schema Gaps)

### 3.1 import_rules — 输出结构未定义

| 来源 | 输出形状 |
|------|----------|
| **OpenAPI** | `type: object` (无具体属性) |
| **实现** | 根据导入方式不同输出不同结构：<br>- **presetId**: `{ preset, presetName, decisions, total }`<br>- **exportJson/filePath**: `{ decisions, total, warnings }` from `io.importRules()`<br>- **错误**: `{ error: { code, message, recoverable } }` |

**影响**: OpenAPI 输出定义过于模糊，无法根据它生成类型正确的客户端。

### 3.2 list_rule_presets — RulePreset 缺少 description

| 来源 | RulePreset 字段 |
|------|-----------------|
| **OpenAPI** | `id`, `name`, `techStack`, `decisions[]` |
| **实现** | `id`, `name`, **`description`**, `techStack`, `decisions[]` |

**差异**: 实现中每个 preset 都有 `description` 字段，OpenAPI 未定义。

### 3.3 list_rule_exports — exports 类型不匹配

| 来源 | exports 元素类型 |
|------|------------------|
| **OpenAPI** | `items: type: object`（数组元素为 object） |
| **实现** | `items: string[]`（数组元素为文件名 string） |

**差异**: 实现返回的是 `.json` 文件名列表（strings），OpenAPI 描述为 objects。

### 3.4 collect_ab_metrics — 输出结构未定义

| 来源 | 输出形状 |
|------|----------|
| **OpenAPI** | `type: object` (无具体属性) |
| **实现** | `{ collected: boolean, message: string }` |

**影响**: 与 3.1 同理，输出 schema 太模糊。

### 3.5 suggest_error_improvement — EvaluationStats 额外字段

| 来源 | EvaluationStats 字段 |
|------|---------------------|
| **OpenAPI** | `totalRecords`, `templateStats` |
| **实现** | `totalRecords`, `ratedRecords`, `averageRating`, `helpfulRate`, `followRate`, `templateStats` |

**差异**: 实现返回 4 个 OpenAPI 未定义的额外字段（`ratedRecords`, `averageRating`, `helpfulRate`, `followRate`）。

### 3.6 start_ab_test — 输出含 message 字段

| 来源 | 输出字段 |
|------|----------|
| **OpenAPI** | `testId`, `status`, `startTime`, `endTime` |
| **实现** | `testId`, `status`, `startTime`, `endTime?`, **`message`** |

**差异**: 实现返回 `message` 字段，OpenAPI 未定义。另外 `endTime` 在实现中 optional，OpenAPI 无此标记。

### 3.7 rollback — 空数组 vs absent 字段

| 来源 | failed / errors 为空时 |
|------|----------------------|
| **OpenAPI** | `nullable: true` |
| **实现** | 返回 `undefined`（JSON 中字段消失），而非 `null` |

**差异**: 当没有失败/错误时，OpenAPI 期望 `null`，但实际 JSON 中这些字段完全不存在。

## 4. 响应格式差异 (Response Format)

### 4.1 MCP 消息包装

**这是一个系统性差异**，因为 MCP 协议特性而非实现选择：

- **OpenAPI 描述**: 直接返回结构化 JSON（如 `{ decisions: [...], conflicts: [...] }`）
- **实际响应**: 包裹在 MCP 响应格式中：
  ```json
  {
    "content": [
      {
        "type": "text",
        "text": "{ \"decisions\": [...], \"conflicts\": [...] }"
      }
    ]
  }
  ```

**影响**: OpenAPI 的 `responses` 定义描述的是 "unwrapped" JSON payload，不是 MCP 线缆协议的实际格式。如果生成客户端代码，需要额外处理 MCP 协议层。

### 4.2 错误响应格式

| 来源 | 错误时返回 |
|------|-----------|
| **OpenAPI** | `HarnessError` schema（某些路径定义了 400 响应） |
| **实现** | MCP `isError: true` 标记，内容为 `JSON.stringify({ error: { code, message, detail, recoverable } })` |

**差异**: 
- OpenAPI 对大多数工具只定义了 200 响应（evaluate_rules/generate_config/analyze_rule_adjustments/export_rules/import_rules 有 400 定义，其余没有）
- 实现中错误始终通过 `isError: true` + `{ error: HarnessError }` 返回
- 实际使用的 `HarnessError` 比 OpenAPI 错误响应的场景更广泛

## 5. Schema / Type 级别差异

### 5.1 SuitabilityWarning.type 枚举

| 来源 | 枚举值 | 实现实际使用 |
|------|--------|-------------|
| **OpenAPI** | `prototype`, `script`, `cost`, `overhead` | `prototype`, `script`, `overhead` |
| **OpenAPI 描述** | "实现使用 script/cost/overhead" | 实际使用 prototype/script/overhead |

**差异**: 
- OpenAPI 的描述文字说实现使用 `script/cost/overhead`，但实际实现使用 `prototype/script/overhead`（`cost` 从未被使用，`prototype` 一直使用）
- 描述与实现正好说反了

### 5.2 RuleDecision 的 adjustedCost / feedbackSpeed / evidence

| 字段 | OpenAPI | 实现 |
|------|---------|------|
| adjustedCost | optional | ✅ 由 engine._decide() 返回 |
| feedbackSpeed | optional | ✅ 由 engine._decide() 返回 |
| evidence | optional | ✅ 通过 `(rule as any).evidence` 传递 |

**差异**: 无功能差异。但 `evidence` 字段在 types.ts 的 `RuleDefinition` 接口中不存在，engine 中通过 `as any` 访问。如果 `rules.json` 不含 `evidence`，返回 `undefined`。

### 5.3 HarnessState 的 validation.findings 字段

| 来源 | 计算方式 |
|------|----------|
| **OpenAPI** | `findings: type: integer` |
| **实现 (state.ts:144)** | `findings = errors + warnings + info` (来自 validate_setup 的结果) |

**差异**: 无 — 实现与定义一致。但在 `QueryStateOutput` 中 `validation.findings` 定义一致。

### 5.4 ScanSuggestion — description 字段

| 来源 | 值 |
|------|-----|
| **OpenAPI** | `description: type: string` |
| **实现 (code_scanner.ts:370)** | `"${ruleId} — found ${hits.length} occurrence(s)"` |

**差异**: 无结构性差异，但实现使用固定格式的描述文字。

## 6. 行为 / 实现细节差异

### 6.1 generate_config — summary.created 计数逻辑

**OpenAPI**: `ConfigFileResult.action` 枚举包括 `[create, overwritten, skipped, merged, dry_run]`

**实现 (index.ts:395)**: 
```
summary.created = files.filter(f => action === "create" || action === "overwritten" || action === "merged").length
summary.updated = files.filter(f => action === "overwritten").length
```

- `created` 将 `overwritten` 也计入，这在语义上模糊（"created" 和 "overwritten" 含义不同）。
- `updated` 和 `created` 有重叠（overwritten 文件被双边计数）。

### 6.2 scan_codebase — empty conflicts

**实现 (index.ts:519-520)**: 保存到 state 时使用硬编码 `conflicts: []`，而非运行冲突检测。

虽然不是直接的协议差异，但 `config 执行扫描后，保存的 `engineOutput` 没有冲突信息。后续 `get_rule_stats` 如果检查冲突会得到空数组。

### 6.3 cognitive_skill — 错误响应格式

**OpenAPI**: `200` 返回 `CognitiveResponse`（oneOf 三种结果）
**实现 (index.ts:1220-1227)**: 当 rule 未找到时，抛出 Error 并返回 `{ error: HarnessError }`，使用 `isError: true`

虽然这是合理的错误处理，但 OpenAPI 没有定义此错误路径（没有对应的 `400` 响应定义）。

### 6.4 generateProjectFiles — 只生成核心 4 种文件

**实现 (index.ts:1363-1389)**: `generateProjectFiles()` 只生成 CLAUDE.md + ESLint + settings.json + .gitignore 四种文件。

**OpenAPI**: generate_config 的文档描述 "生成 CLAUDE.md、ESLint 配置、settings.json、.gitignore 等文件"，implicitly 暗示可能生成更多。

注意: `init_harness` (index.ts:596-613) 会额外生成 Husky hooks 和 CI workflow，但 `generate_config` 不会。这与 OpenAPI 描述一致（generate_config 的 scope 只有核心文件）。

## 8. 总结与严重性评级

### 高 (High) — 可能影响客户端兼容性

| # | 差异 | 影响 |
|---|------|------|
| H1 | **init_harness 输入必填性不一致**: OAS 声明 `projectPhase/teamSize/techStack` 可选，Zod 必填 | 按 OAS 生成的客户端调用将失败 |
| H2 | **list_rule_exports 输出类型不匹配**: OAS 说 `object[]`，实现返回 `string[]` | 类型不兼容 |
| H3 | **import_rules/collect_ab_metrics 输出未定义**: OAS 为 `type: object` | 无法生成类型安全的客户端 |

### 中 (Medium) — 文档准确性问题

| # | 差异 | 影响 |
|---|------|------|
| M1 | **SuitabilityWarning.type 描述与实现相反**: OAS 说"实现使用 script/cost/overhead"，实际使用 prototype/script/overhead | 文档误导 |
| M2 | **EvaluationStats 额外字段**: 实现返回 4 个 OAS 未定义的字段 | 文档不完整 |
| M3 | **RulePreset 缺少 description**: OAS 未包含此字段 | 文档不完整 |
| M4 | **rollback 空字段行为**: OAS 说 nullable，实现返回 undefined（消失） | 实现偏差 |
| M5 | **start_ab_test 输出缺 message**: 实现返回 message，OAS 未定义 | 文档不完整 |

### 低 (Low) — 小问题或文档风格

| # | 差异 | 影响 |
|---|------|------|
| L1 | **import_rules 三选一约束**: 仅在运行时检查，Schema 层未体现 | 轻微 |
| L2 | **generate_config created/updated 重叠计数**: `created` 包含 `overwritten` | 语义模糊 |
| L3 | **scan_codebase 硬编码空 conflicts**: 扫描后 state 中 conflicts 始终为空 | 功能性缺失 |
| L4 | **MCP 响应包装格式**: OAS 描述 raw JSON，实际为 `{content: [{text}]}` | 协议层特性 |
| L5 | **cognitive_skill 错误响应未定义**: rule 未找到时无 OAS 错误路径定义 | 遗漏 |

## 9. 推荐修复

### 建议更新 OpenAPI 文档
1. 修复 `init_harness` 的 `projectPhase/teamSize/techStack` 为 required
2. 为 `import_rules` 输出定义具体结构
3. 为 `collect_ab_metrics` 输出定义具体结构
4. `list_rule_exports` 的 `exports` 改为 `type: string`
5. `RulePreset` 增加 `description` 字段
6. `EvaluationStats` 补充 `ratedRecords/averageRating/helpfulRate/followRate` 字段
7. 更正 `SuitabilityWarning.type` 的描述文字
8. `start_ab_test` 输出增加 `message` 字段
9. `rollback` 输出的 `failed/errors` 改为非 nullable（或实现返回 null）
10. 在认知层工具中定义 400 错误响应

### 建议更新实现
11. `init_harness` Zod schema — 考虑将 `projectPhase/teamSize/techStack` 改为 optional（与 OAS 对齐，支持纯 preset 调用）
12. `scan_codebase` — 在保存到 state 前运行冲突检测
13. `rollback` — 将 `failed/errors` 设为 `null` 而非消除字段
