## GAP 1: HarnessState.validation 引用错误 Schema

**Severity:** GAP

**Spec:** `openapi.yaml:2178-2179`
```yaml
validation:
  $ref: "#/components/schemas/ValidateSetupOutput"
```

**Implementation:** `types.ts:236-242`
```typescript
validation?: {
  status: "pass" | "warn" | "fail";
  errors: number;
  warnings: number;
  findings: number;
  checkedAt: string;
}
```

**Problem:** HarnessState.validation 引用 `ValidateSetupOutput`（包含 `projectDir`、`findings: ValidationFinding[]`、含 `passed` 的 `summary`、`nextSteps: string[]` 等字段），但实际存储的是轻量摘要（5 个字段）。规范暗示整个验证输出被持久化，与实际存储不匹配。

对比 `QueryStateOutput`（`openapi.yaml:1575-1589`）——它在内联属性中正确定义了摘要子集，与实现一致。但 `HarnessState` 复用了完整的 `ValidateSetupOutput` 引用，产生了误导。

**Fix:** 在 `HarnessState` 中将 `validation` 改为内联摘要类型（匹配 `QueryStateOutput` 的子集），或新建共享类型 `ValidationSummary`。

## GAP 2: init_harness dryRun 返回 `validation: null` 不符合规范

**Severity:** GAP

**Spec:** `openapi.yaml:243-252`
```yaml
summary:
  $ref: "#/components/schemas/InitHarnessSummary"
validation:
  $ref: "#/components/schemas/ValidateSetupOutput"   # no nullable: true
```

**Implementation:** `index.ts:491-506`
```typescript
if (input.dryRun) {
  return {
    content: [{
      type: "text",
      text: JSON.stringify({
        files: [],
        summary: {
          files: { total: 0, created: 0, updated: 0, skipped: 0 },
          decisions: evalOutput.summary.total,
        },
        validation: null,   // <-- null literal, but schema doesn't allow null
      }, null, 2),
    }],
  };
}
```

**Problem:** dryRun 路径返回 `validation: null`，但规范引用的是 `ValidateSetupOutput`（对象类型），未设置 `nullable: true`，因此 `null` 在规范中无效。

此外，dryRun 的 `summary` 对象缺少 `backupDir`、`recommendedActions`、`hubSetup` 字段（均在 `InitHarnessSummary` 中定义）。这些字段不在 `required` 列表中，技术上合法，但与规范产生分歧。

**Fix:** 选项 (a) 从 dryRun 响应中省略 `validation`，或 (b) 为规范中的 `validation` 添加 `nullable: true`。

## MINOR 4: 多处 integer / number 类型不一致

**Severity:** MINOR

在以下字段中，规范声明 `type: integer` 但 TypeScript 使用 `number`（接受浮点数）：

| Schema | Field | Spec Line | TS Location | TS Type |
|--------|-------|-----------|-------------|---------|
| `RuleDecision` | `adjustedCost` | 1465 | `types.ts:115` | `number` |
| `RuleDecision` | `feedbackSpeed` | 1468 | `types.ts:117` | `number` |
| `SuitabilityAssessment` | `score` | 1745 | `types.ts:262` | `number` |
| `OptimizeErrorMessageInput` | `lineNumber` | 932 | `types.ts:489` | `number` |

**Problem:** 运行时值确实是整数（来自 `Math.round()` 或整数算术），但 TypeScript 类型允许浮点数。规范生成的客户端可能期望严格整数并执行类型检查。

**Fix:** 在类型安全一端修复——对声明为整数的字段使用 `z.number().int()`，或对不严格要求整数的字段将规范改为 `type: number`。

## MINOR 5: HarnessStatus 枚举包含冗余的 null

**Severity:** MINOR

**Spec:** `openapi.yaml:999-1003`
```yaml
HarnessStatus:
  type: string
  nullable: true
  enum: [null, evaluated, confirmed, generated, validated]
```

**Problem:** `nullable: true` 已经允许 JSON null 值。枚举中同时包含 `null` 字符串值与 `nullable: true` 配合 `type: string` 会产生歧义：`null` 在枚举中解释为 JSON null 还是字符串 `"null"`？实现（`types.ts:34-39`）使用联合类型 `| null`，不会产生此歧义。

**Fix:** 移除枚举中的 `null` 值，仅保留 `nullable: true`：
```yaml
enum: [evaluated, confirmed, generated, validated]
```

## MINOR 6: `hubSetup` 字段名疑似拼写错误

**Severity:** MINOR

**Spec:** `openapi.yaml:2218`
```yaml
hubSetup:
  type: string
  nullable: true
  description: Husky 安装指引
```

**Implementation:** `index.ts:587`（使用相同名称 `hubSetup`）

**Problem:** 名称 `hubSetup` 疑似 `huskySetup` 的拼写错误（缺少字母 "s"）。规范描述为 "Husky 安装指引" 证实该字段指向 Husky，非 GitHub Hub。虽然实现与规范保持一致（都使用了错误拼写），但该命名不合理。

**Fix:** 在 `openapi.yaml` 和 `index.ts` 中将 `hubSetup` 重命名为 `huskySetup`。

## MINOR 7: engine.ts 通过 `as any` 访问 RuleDefinition 上不存在的 `evidence` 字段

**Severity:** MINOR

**Spec:** `RuleDecision.evidence` 在 `openapi.yaml:1470-1472` 定义：
```yaml
evidence:
  type: string
  description: 设计 §3.1.1：决策支持证据
```

**Implementation:** `engine.ts:278`
```typescript
evidence: (rule as any).evidence,
```

**Problem:** `RuleDefinition`（`types.ts:80-99`）**没有** `evidence` 字段。`as any` 强制转换抑制了 TypeScript 编译错误，但运行时值始终是 `undefined`。这意味着引擎输出的每个 `RuleDecision.evidence` 始终为 `undefined`——该字段在语义上被破坏。

`RuleDecision.evidence` 被记录为 "决策支持证据"，暗示应从规则定义或某些上下文填充，但从未实现。

**Fix:** 选项 (a) 为 `RuleDefinition` 添加 `evidence` 字段并从 `rules.json` 填充数据，或 (b) 移除 `engine.ts:278` 的赋值并从 `RuleDecision` 删除该字段。
