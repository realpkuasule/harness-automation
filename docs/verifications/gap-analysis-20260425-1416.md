## 严重差异 (Critical)

### C1. validate_setup 输出结构不一致

**OpenAPI 描述** (`ValidateSetupOutput`):
```yaml
properties:
  status: string[pass/warn/fail]
  checks: array[ValidationCheck]    # { name, status, message, fixCommand }
  nextSteps: array[string]
```

**实际实现** (`setup_validator.ts` — `ValidationResult`):
```typescript
{
  projectDir: string;
  findings: ValidationFinding[];    // { file, type: error|warning|info, message, fix? }
  summary: { errors, warnings, info, passed, status };
  nextSteps: string[];
}
```

**差异**:
- OpenAPI 使用 `checks` 字段，实现使用 `findings` 字段（命名和类型均不同）
- OpenAPI 将 `status` 放在顶层，实现将其嵌套在 `summary` 对象内
- OpenAPI `ValidationCheck` 有 `name`/`fixCommand`，实现 `ValidationFinding` 有 `file`/`fix?`
- 实现额外返回 `projectDir`、`summary.errors/warnings/info/passed`，OpenAPI 未描述

**影响**: 调用方若按 OpenAPI 解析 validate_setup 返回结果，将无法正确读取。

### C2. scan_codebase 输出结构不一致

```yaml
properties:
```

- 扫描发现嵌套在 `scanSummary.suggestions` 中，不在顶层
- 缺少 `existingRules` 独立数组（实现合并入 extraDecisions）

**影响**: 调用方按 OpenAPI 期望的结构无法匹配实现返回值。

### C5. MEDIUM_STRICTNESS 缺失三个介质值

**文件**: `rule_adapter.ts:47`
```typescript
const MEDIUM_STRICTNESS: RuleMedium[] = ["claude_md", "settings", "linter_warn", "hook", "ci"];
```

**OpenAPI 定义** (`RuleMedium` 枚举):

**影响**: analyze_rule_adjustments 对有这些介质的规则静默跳过，不报错不推荐。

**位置**: `docs/api/openapi.yaml:1562-1564`
```yaml

  type: object
```
