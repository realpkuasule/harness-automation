## P1 Items

### P1-1: `query_state.lastEvalAt` Drifts After Non-Eval State Transitions

**History**: New finding (first appeared in 2002 report, §3.2)
**Source**: `gap-analysis-20260425-2002.md §3.2`
**Recurring**: No

**Root cause**: `index.ts:376` sets `result.lastEvalAt = state.updatedAt`, but `updatedAt` is overwritten by **every** `StateManager.save()` call — including `confirm_decisions`, `setProjectInfo`, `logGeneration`, and `setValidation`. After any of these, `lastEvalAt` reflects the most recent save time rather than the evaluation time.

**Sequence demonstrating the drift**:
1. `evaluate_rules` → state save sets `updatedAt = T1` (eval time) ✅
2. `confirm_decisions` → state save sets `updatedAt = T2` (confirm time, T2 > T1)
3. `query_state` → returns `lastEvalAt = T2` (wrong — should be T1)

**Files to modify**:

| # | File | Line(s) | Change |
|---|------|---------|--------|
| 1a | `mcp-server/src/types.ts` | ~221 (HarnessState) | Add optional field: `evaluatedAt?: string` |
| 1b | `mcp-server/src/state.ts` | 79-85 (setEngineOutput) | Add `state.evaluatedAt = new Date().toISOString()` alongside existing `phase` advancement |
| 1c | `mcp-server/src/state.ts` | 69-76 (setEngineInput) | Same: add `state.evaluatedAt` assignment (input + output both represent "evaluation time") |
| 1d | `mcp-server/src/index.ts` | 375-377 (query_state handler) | Change `result.lastEvalAt = state.updatedAt` → `result.lastEvalAt = state.evaluatedAt ?? state.updatedAt` (fallback for existing states without `evaluatedAt`) |

**Verification**:
1. Run `npx vitest run` — all tests must pass
2. Manually trace: call `evaluate_rules` → `confirm_decisions` → `query_state`, verify `lastEvalAt` matches eval time, not confirm time
3. Confirm backward compat: old state files without `evaluatedAt` field still report `lastEvalAt` correctly (via fallback to `updatedAt`)

### P1-2: `CognitiveAutoTrigger` Return Type Too Broad

**History**: New finding (first appeared in 2002 report, §3.1)
**Source**: `gap-analysis-20260425-2002.md §3.1`
**Recurring**: No

**Root cause**: `shouldAutoTrigger()` returns `CognitiveRequest | null`, where `CognitiveRequest` is an 11+-field interface. The spec defines `CognitiveAutoTrigger` as a constrained 4-field schema. Although the runtime output matches today (only 4 fields populated), the type system allows accidental field leakage if `shouldAutoTrigger` is later modified.

**Files to modify**:

| # | File | Line(s) | Change |
|---|------|---------|--------|
| 2a | `mcp-server/src/types.ts` | (new, after `ErrorMessage` block) | Add dedicated `CognitiveAutoTrigger` interface with exactly 4 fields: `skillType: "educational"`, `ruleId: string`, `topic: string`, `experienceLevel: "beginner" \| "intermediate" \| "advanced"` |
| 2b | `mcp-server/src/cognitive_layer/orchestrator.ts` | 89-91 (shouldAutoTrigger signature) | Change return type from `CognitiveRequest | null` to `CognitiveAutoTrigger | null` (after importing) |
| 2c | `mcp-server/src/index.ts` | 1191 (repeatedPattern variable) | Change type annotation from `CognitiveRequest` to `CognitiveAutoTrigger` |

**Verification**:
1. Run `npx vitest run` — all tests must pass
2. Confirm TypeScript compilation: `npx tsc --noEmit` — no type errors
3. Verify `shouldAutoTrigger` usage in `skill-workflow.test.ts` still passes (it destructures `repeatedPattern`)

### P2-2: `init_harness` Preset Description Misleading

**History**: New finding (first appeared in 2002 report, §3.4)
**Source**: `gap-analysis-20260425-2002.md §3.4`
**Recurring**: No

**Root cause**: `InitHarnessInput` in the spec describes `preset` as "嵌套 preset 对象（实现新增，支持简化调用）". The "支持简化调用" (supports simplified calling) is misleading because `projectPhase`, `teamSize`, and `techStack` are still required flat fields; Zod rejects input without them. The preset is actually a merge/fallback mechanism where flat params take precedence.

**Files to modify**:

| # | File | Line(s) | Change |
|---|------|---------|--------|
| 4a | `docs/api/openapi.yaml` | 1238 (preset description) | Change "嵌套 preset 对象（实现新增，支持简化调用）" → "嵌套 preset 对象，作为扁平参数的替代嵌套写法。扁平参数优先级高于 preset。使用 preset 时仍需提供 projectPhase、teamSize、techStack（或通过 preset 填充）。" |

**Verification**:
1. Read the modified spec line — description now accurately reflects behavior
2. No code change needed

**Source**: Cross-reference between `CognitiveAutoTrigger` schema and `cognitive_skill` tool spec

An explicit `$ref` from `CognitiveAutoTrigger` to `CognitiveSkillInput` would make the connection machine-readable.

**Verification**:
1. Read the modified spec — connection between the two schemas is now explicit in description
2. No code change needed

### P2-4: `HarnessState.validatedAt` Missing from Spec

**History**: New finding (missed in prior reports)
**Source**: Spec `HarnessState` schema vs impl `HarnessState` interface

**Root cause**: The implementation's `HarnessState` interface (`types.ts:234`) has `validatedAt?: string`, but the OpenAPI spec's `HarnessState` schema does not include this field. The spec also shows `validatedAt` in `QueryStateOutput` (which is separately defined), but not in `HarnessState` itself. This means `HarnessState` in the spec is missing a field that can appear in the state file.

**Files to modify**:

### P2-5: `HarnessState.phase` Nullable Inconsistency

**History**: Recurring from pre-P13 era (noted in spec but inconsistent representation)
**Source**: Spec `HarnessStatus` vs `HarnessState.phase`

**Root cause**: The spec defines `HarnessStatus` as `nullable: true` with `enum: [evaluated, confirmed, generated, validated]`. But `HarnessState.phase` uses `$ref: "#/components/schemas/HarnessStatus"` — the `nullable: true` is on the `HarnessStatus` enum definition, not on the `phase` property reference. In JSON Schema, `nullable` on the enum type might not propagate correctly to referencing properties depending on the tooling.

**Files to modify**:

| # | File | Line(s) | Change |
|---|------|---------|--------|
| 7a | `docs/api/openapi.yaml` | ~2224 (HarnessState.phase) | Change `$ref` to inline: add `nullable: true` alongside the ref, or convert to explicit `oneOf` with `null` and enum string. Preferred approach: keep `$ref` but add explicit comment |

Actually this is very minor and tooling-dependent. Let me drop it — too pedantic.
