## P0-1: enrichPartialDecisions 返回占位符描述

### 来源
verification-20260427-2306.md 问题 1

### 当前状态验证

```javascript
// mcp-server/src/index.ts:1275-1298
function enrichPartialDecisions(partials): RuleDecision[] {
  const rules = loadRules();
  return partials.map((p) => {
    const rule = rules.find((r) => r.id === p.ruleId);
    // ...
    return {
      ruleId: rule.id,
      ruleName: rule.name,
      recommendedMedium: medium,
      alternativeMedia: rule.alternativeMedium,
      confidence: 0.8,
      reasons: ["User-configured medium assignment"],  // ← 占位符
      cognitiveLayerRequired: false,
      cognitiveSkillTriggers: [],
    };
  });
}
```

`generate_config` 在接收用户传入的精简决策 `{ruleId, recommendedMedium}` 时调用此函数。但 `reasons` 固定为占位符，**未从 rule 定义读取** `formalizable`、`cost`、`frequency` 等属性来构建实义描述。同时 `errorMessage`、`adjustedCost`、`adjustedCostLabel`、`feedbackSpeed` 字段也丢失了。

而 `init_harness` 直接使用 `DecisionEngine.evaluate()` 输出，包含完整的 `reasons`（如 `"规则可形式化，适合自动化检查"`、`"实施成本低 (1/5)"`），产生文本质量差异。

### 改动

| 文件 | 行号 | 当前内容 | 改为 | 说明 |
|------|------|----------|------|------|
| `mcp-server/src/index.ts` | L1275-1298 | `reasons: ["User-configured medium assignment"]` + 缺失 errorMessage/adjustedCost 等字段 | 从 rule 定义读取属性构建实义 reasons，补充所有缺失字段 | enrich 完整决策 |

具体实现：

```typescript
// 从 rule 定义构建实义 reasons
const reasons: string[] = [];
if (rule.formalizable) {
  reasons.push("规则可形式化，适合自动化检查");
} else {
  reasons.push("规则不可完全形式化，需要认知层支持");
}
reasons.push(`实施成本 ${rule.cost === 1 ? "低" : rule.cost <= 3 ? "中" : "高"} (${rule.cost}/5)`);
if (rule.frequency <= 2) {
  reasons.push(`触发频率低 (${rule.frequency}/5)`);
} else if (rule.frequency >= 4) {
  reasons.push(`触发频率高 (${rule.frequency}/5)`);
}
if (!rule.formalizable) {
  reasons.push("建议配合认知层 Skills 使用");
}

return {
  ...rule,
  ruleId: rule.id,
  ruleName: rule.name,
  recommendedMedium: medium,
  alternativeMedia: rule.alternativeMedium,
  confidence: 0.8,
  reasons,
  cognitiveLayerRequired: !rule.formalizable,
  cognitiveSkillTriggers: !rule.formalizable
    ? (rule.cognitiveLayerSupport?.skillTriggers ?? ["diagnostic", "educational"])
    : [],
  adjustedCost: rule.cost,
  adjustedCostLabel: rule.cost <= 2 ? "low" : rule.cost <= 3 ? "medium" : "high",
  feedbackSpeed: rule.feedbackSpeed,
  errorMessage: rule.errorMessage,
};
```

### 验证方法

1. 调用 `generate_config` 传入精简决策 `[{ruleId: "R001", recommendedMedium: "linter_warn"}]`，返回的 CLAUDE.md 中 no-console-log 描述包含实义文本（非 "User-configured medium assignment"）
2. 现有测试中 `index.test.ts` 的 enrich 相关测试更新断言
3. `npm run test` — 全部通过

## P1-1: eslint.config.js 仅输出 CommonJS，不支持 ESM 项目

### 来源
verification-20260427-2306.md 问题 2

### 当前状态验证

```javascript
// mcp-server/src/generators/eslint.ts:71
return `module.exports = ${JSON.stringify(configArray, null, 2)};\n`;
```

始终输出 CommonJS 格式。当项目使用 `"type": "module"` 时，Node.js 会将 `.js` 文件视为 ESM，导致 `module.exports` 语法错误。

### 改动

| 文件 | 行号 | 当前内容 | 改为 | 说明 |
|------|------|----------|------|------|
| `mcp-server/src/generators/eslint.ts` | L71 | 硬编码 CommonJS 模板 | 动态判断项目 `package.json` 的 `"type"` 字段，ESM 用 `export default`，CJS 用 `module.exports` | 格式自适应 |
| `mcp-server/src/generators/eslint.ts` | 函数签名 | `generateEslintConfig(config: EslintConfig)` | 新增 `projectDir` 可选参数 | 传入项目路径用于读取 package.json |
| `mcp-server/src/generators/eslint.test.ts` | 添加用例 | — | 新增 `"type": "module"` 场景的测试 | 验证 ESM 格式输出 |
| `docs/api/openapi.yaml` | L74-77 | `四种核心配置文件` + `module.exports = [...]` | `核心配置文件` + `Flat Config 格式（根据项目 type 字段自动选择 CommonJS 或 ESM）` | generate_config 描述更新 |

具体实现：

```typescript
export function generateEslintConfig(config: EslintConfig & { projectDir?: string }): string {
  // ... 现有逻辑构建 configArray ...

  // 检测项目是否为 ESM
  let isEsm = false;
  if (config.projectDir) {
    try {
      const pkgPath = join(config.projectDir, "package.json");
      const pkg = JSON.parse(readFileSync(pkgPath, "utf-8"));
      isEsm = pkg.type === "module";
    } catch { /* 默认 CJS */ }
  }

  const configJson = JSON.stringify(configArray, null, 2);
  return isEsm
    ? `export default ${configJson};\n`
    : `module.exports = ${configJson};\n`;
}
```

### 验证方法

1. 测试 A：项目无 `"type": "module"` → 输出 `module.exports = [...]`
2. 测试 B：项目有 `"type": "module"` → 输出 `export default [...]`
3. 测试 C：项目无 package.json → 默认输出 `module.exports = [...]`
4. ESM 输出可直接被 `node --input-type=module` 加载验证
