## P1 — 类型精度问题（重要，影响客户端类型检查）

### P6-24: 修正 integer/number 类型一致性 [MINOR 4]

| 项目 | 内容 |
|------|------|
| **类型** | 首次出现 |
| **问题** | 以下字段 OpenAPI 声明 `type: integer` 但 TypeScript 使用 `number`（未约束整数）： |
| | 1. `RuleDecision.adjustedCost` — `openapi.yaml:1465` → `types.ts:115` |
| | 2. `RuleDecision.feedbackSpeed` — `openapi.yaml:1468` → `types.ts:117` |
| | 3. `SuitabilityAssessment.score` — `openapi.yaml:1745` → `types.ts:262` |
| | 4. `OptimizeErrorMessageInput.lineNumber` — `openapi.yaml:932` → `types.ts:489` |
| **文件 & 行号** | 参考上表对应位置 |
| **具体改动** | **方案 A（推荐——从代码侧修复，精度最高）**：在涉及 Zod schema 的地方（如 `lineNumber`）使用 `z.number().int()`；对纯 interface 字段保持`number`不变，但更新 **OpenAPI YAML 为 `type: number`**——因为 TypeScript 不保证 `Math.round()` 调用处的类型安全，改为 number 更诚实。 |
| | 具体：`openapi.yaml:1465` `type: integer` → `type: number`；`openapi.yaml:1468` `type: integer` → `type: number`；`openapi.yaml:1745` `type: integer` → `type: number`；`openapi.yaml:932` `type: integer` → `type: number`。 |
| **验证方法** | `openapi.yaml` 格式校验；确认 4 处字段值类型改为 `number` |

### P6-25: 修正 HarnessStatus 枚举冗余 null [MINOR 5]

| 项目 | 内容 |
|------|------|
| **类型** | 首次出现 |
| **问题** | `openapi.yaml:999-1003` 中 `HarnessStatus` 同时使用 `nullable: true` 和 `enum: [null, evaluated, confirmed, generated, validated]`。`null` 在枚举中与 `type: string` 配合使用可能产生歧义（JSON null vs 字符串 "null"）。实现（`types.ts:34-39`）使用 `| null` 联合类型，无此问题。 |
| **文件 & 行号** | `docs/api/openapi.yaml:1002` |
| **具体改动** | 移除 `enum` 中的 `null` 值，仅保留 `nullable: true`：`enum: [evaluated, confirmed, generated, validated]` |
| **验证方法** | `openapi.yaml` 格式校验 |
