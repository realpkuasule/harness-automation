## 一、文件覆盖/合并策略 (CRITICAL — 4 项)

### P1-1 `.gitignore` 全量覆盖 → 改为真合并

**现象**: `init_harness`/`generate_config` 写入 `.gitignore` 时，action 标记为 `merged`，但实际行为是**全量替换**。目标项目原有的 63 行 `.gitignore`（Python/Node/IDE/Docker/监控数据等）被替换为仅 4 行的 harness 条目。用户只能通过 `git checkout -- .gitignore` 恢复原始文件后手动追加。

**根因**: `src/index.ts:generateProjectFiles()` 调用 `writeFileSync` 写入 `.gitignore`，但生成器 `gitignore.ts` 只产出 harness 条目，调用方未将新条目**追加**到现有文件末尾。

**修复方案**:
1. `src/generators/gitignore.ts` — 已是正确的（只产出增量条目），无需改动
2. `src/index.ts:generateProjectFiles()` L1374-1378 — 对 `.gitignore` 使用**读取现有→追加→合并写入**的逻辑，而非直接覆盖 `writeFileSync`
3. 调用方需读取现有 `.gitignore` 内容，传入 `generateGitignore(existingContent)`，将返回内容追加到原文件末尾

**涉及文件**: `mcp-server/src/index.ts` (L1374-1378), `mcp-server/src/generators/gitignore.ts`

---

### P1-2 `.github/workflows/ci.yml` 未加入备份列表

**现象**: 目标项目原有的 CI workflow（`name: CI - Build & Quality`）被 harness 生成的 CI 配置直接覆盖，且**未出现在备份中**。`backupGeneratedFiles()` 的 candidates 列表包含了 `.github/workflows/ci.yml`（L1406），但实际运行中备份目录只包含 `CLAUDE.md` 和 `eslint.config.js`，`ci.yml` 未被备份。

**根因**: 需要核实 `backupGeneratedFiles()` L1400-1407 — candidates 列表包含 `.github/workflows/ci.yml`，但 `existsSync` 检查可能因路径问题跳过。也可能是两次 `init_harness` 调用之间的状态不一致。

**修复方案**:
1. 确认 `backupGeneratedFiles()` 中路径正确，增加日志记录哪些文件被备份
2. 对 CI workflow 文件采取 **merge 策略**而非直接覆盖：检测到已有 CI 文件时，将 harness 生成的 CI 步骤**追加**到现有 workflow 中
3. 在 CI 生成前提示用户已有 CI 配置将被修改

**涉及文件**: `mcp-server/src/index.ts` (L1399-1424), `mcp-server/src/generators/ci.ts`

---

### P1-3 `package.json` 全量创建 → 改为智能合并

**现象**: 目标项目是 monorepo（无根 `package.json`），harness 创建了一个新 `package.json`。但如果目标项目已有 `package.json`，当前逻辑会直接覆盖。此外，`packageJson` backup 为 `null`（L200-201），无法回滚。

**根因**: `mergeDependencies()` 函数只处理 devDependencies 的逻辑合并，但 `init_harness` 的流程中并未使用其合并能力——它直接用 `writeFileSync` 覆盖整个文件。

**修复方案**:
1. 在写入 `package.json` 前检查文件是否存在
1. 若存在：读取现有内容 → 合并 scripts、devDependencies、lint-staged 字段 → 写回
2. 若不存在：创建最小化 `package.json`（仅含 harness 需要的字段）
3. `backupGeneratedFiles()` 增加 `package.json` 到 candidates

**涉及文件**: `mcp-server/src/index.ts`, `mcp-server/src/generators/package_json.ts`

---

### P1-4 `.lintstagedrc.json` 与 `package.json` 配置重复

**现象**: harness 同时生成 `.lintstagedrc.json` 文件和 `package.json` 中的 `lint-staged` 字段，产生冗余配置。用户需手动删除 `.lintstagedrc.json`。

**根因**: `generateProjectFiles()` 调用 `generateLintStagedConfig()` 生成独立文件，同时 `mergeDependencies()` 也会在 package.json 中添加 lint-staged 配置。

**修复方案**:
- 统一策略：**优先使用 `package.json` 中的 `lint-staged` 字段**（更标准、减少文件数量）
- 移除 `.lintstagedrc.json` 生成，改为在 package.json 合并逻辑中追加 lint-staged 配置
- 如果项目已有 `.lintstagedrc.json`，则保留原配置并提示用户

**涉及文件**: `mcp-server/src/index.ts` (L1381-1390), `mcp-server/src/generators/husky.ts` (L57-63)

## 二、ESLint 配置生成质量 (HIGH — 3 项)

### P2-1 规则名映射错误：`no-magic-numbers` → `@typescript-eslint/no-magic-numbers`

**现象**: 生成的 ESLint 配置中 `"no-magic-numbers"` 不是 ESLint 内置规则，需要 `@typescript-eslint` 插件前缀。在目标项目中运行 ESLint 时该规则静默失效（插件未加载时无效果）。

**根因**: `src/generators/eslint.ts` L39-41 映射 `"no-magic-numbers"` 规则名未加插件前缀。

**修复方案**:
```typescript
// L39-41 改为:
case "no-magic-numbers":
  rules["@typescript-eslint/no-magic-numbers"] = [severity, { ignore: [0, 1], ignoreEnums: true, ignoreReadonlyClassProperties: true }];
  break;
```

**涉及文件**: `mcp-server/src/generators/eslint.ts` (L39-41)

---

### P2-2 规则 `no-process-env` 不存在于标准 ESLint

**现象**: 生成的 `"no-process-env": ["error"]` 不是 ESLint 内置规则，也不是 `@typescript-eslint` 规则。它来自 `eslint-plugin-node`（已废弃）。在目标项目中该规则完全无效。

**根因**: `src/generators/eslint.ts` L54-55 映射了不存在的规则名。

**修复方案**:
1. 方案 A（推荐）: 从 ESLint 规则中移除 `secure-env-vars` 规则，该规则更适合作为 CLAUDE.md 文档规则或 settings.json 约束
2. 方案 B: 改用 `no-restricted-syntax` 或 `no-restricted-globals` 检测 `process.env` 访问
3. 对应的 rule 定义中 `secure-env-vars` 的推荐介质应从 `linter_error` 改为 `settings` 或 `claude_md`

**涉及文件**: `mcp-server/src/generators/eslint.ts` (L54-55), `mcp-server/src/rules.json`

---

### P2-3 生成的 ESLint 配置缺少插件/解析器声明

**现象**: 生成的 `eslint.config.js` 是一个裸的 rules 对象数组，没有:
- `files` 字段（指定哪些文件应用规则）
- `languageOptions.parser` (TypeScript parser)
- `plugins` 字段（声明 `@typescript-eslint` 插件）

用户（Claude）在目标项目中手动重写了整个配置以添加这些缺失部分（L207）。

**根因**: `src/generators/eslint.ts` L75 生成的 configArray 只包含 `{ rules }`，缺少框架配置。

**修复方案**:
```typescript
// 生成完整的 flat config 结构:
const configArray = [
  ...prepend,
  {
    files: ["**/*.{js,jsx,ts,tsx}"],
    languageOptions: {
      parser: "@typescript-eslint/parser",
      parserOptions: { ecmaVersion: "latest", sourceType: "module" },
    },
    plugins: { "@typescript-eslint": "@typescript-eslint/eslint-plugin" },
    rules,
  },
];
```

同时需要在文件头添加:
```javascript
const tseslint = require("@typescript-eslint/eslint-plugin");
const tsparser = require("@typescript-eslint/parser");
```

**涉及文件**: `mcp-server/src/generators/eslint.ts` (L62-92)

## 三、Toolchain 初始化与状态管理 (MEDIUM — 3 项)

### P3-1 Husky v9 兼容性：hook 中使用已废弃的 `_/husky.sh` 引用

**现象**: 用户 commit 时报错：

```
Error: Exit code 1
     husky - DEPRECATED

     Please remove the following two lines from .husky/pre-commit:

     #!/usr/bin/env sh
     . "$(dirname -- "$0")/_/husky.sh"

     They WILL FAIL in v10.0.0
```

harness 生成的 pre-commit 和 commit-msg hooks 包含了 `. "$(dirname "$0")/_/husky.sh"` 行，这是 Husky v8 时代的机制。Husky v9 已废弃此写法，v10 将直接报错。

**根因**: `generateHuskyConfig()` 在 `husky.ts` L32-37 / L41-47 中生成了已废弃的 `_/husky.sh` 引用:

```javascript
// 旧代码（已废弃）
hooks["pre-commit"] = [
  "#!/bin/sh",
  ". \"$(dirname \"$0\")/_/husky.sh\"",  // ← 将在 v10 失效
  "",
  "npx lint-staged",
].join("\n");
```

Husky v9 的 hook 脚本是纯 shell 脚本，Git 直接执行，不再需要 sourcing `_/husky.sh`。

**修复方案** (已实施):
1. 移除 `pre-commit` 和 `commit-msg` 生成逻辑中的 `. "$(dirname "$0")/_/husky.sh"` 行
2. 修正 `generateHuskySetupInstructions()` — 不再建议用户运行 `npx husky init`（会覆盖 harness hooks），改为建议 `npx husky`（仅设置 `core.hooksPath`）

**涉及文件**: `mcp-server/src/generators/husky.ts` ✅ 已修复

---

### P3-2 `naming-convention` 规则过于激进

**现象**: 目标项目中 ESLint 运行产生 **829 个 warnings**，大部分来自 `naming-convention` 规则（例如 Prisma 生成的 `_count` 属性名）。对于已有代码库，这产生大量噪音。

**根因**: `naming-convention` 规则使用默认配置，未考虑第三方库生成的特殊命名约定（如 Prisma、GraphQL 等）。

**修复方案**:
1. `consistent-naming` 规则的 `naming-convention` 配置增加更多豁免：
   ```typescript
   { selector: "property", format: ["camelCase", "PascalCase", "snake_case", "UPPER_CASE"],
     filter: { regex: "^(_count|_sum|_avg|_min|_max)$", match: false } }
   ```
2. 对于 `growth/mature` 阶段项目，该规则默认级别应为 `warn` 而非 `error`
3. 增加 `argsIgnorePattern: "^_"` 允许下划线前缀

**涉及文件**: `mcp-server/src/generators/eslint.ts` (L45-47), `mcp-server/src/rules.json`

### P3-3 依赖安装耗时过长且无进度反馈

**现象**: `npm install` 耗时 6 分钟完成（242 packages），期间用户只能等待。MCP 工具返回 `backgroundTaskId` 但无实时进度。

**根因**: 这是架构层面的限制，MCP 协议不支持流式进度反馈。

**修复方案**:
1. 在 `init_harness` 输出中添加预估时间提示："Installing dependencies may take 2-5 minutes..."
2. 返回结果中包含清晰的 install 状态日志
3. 考虑将 npm install 拆分为多个小批次（核心依赖优先）

**涉及文件**: `mcp-server/src/index.ts`, `mcp-server/src/generators/package_json.ts`
