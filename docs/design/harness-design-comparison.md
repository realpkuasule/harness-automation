# Harness 设计对比分析

## 概述

本文档对比分析 `harness-automation-design.md`（设计文件）与 `reference.md`（参考文件）之间的差异。参考文件是 Harness Engineering 的方法论文章，设计文件是基于该方法论的自动化系统实现设计。

## 核心概念对比

### 1. 五种介质

**参考文件**：详细介绍了五种介质及其力学特性：
- CLAUDE.md：软约束，灵活但可能漂移
- settings.json：harness 强制行为，确定性高
- 自定义 linter：硬约束，形式化要求高
- Git hook：过程拦截，反馈及时
- CI：终极防线，反馈延迟

**设计文件**：完整实现了五种介质，通过配置生成器自动生成：
- CLAUDE.md 生成器（第8.1节）
- settings.json 生成器（第8.1节）
- ESLint 配置生成器（第8.1节）
- Husky hook 生成器（第8.1节）
- CI 工作流生成器（第8.1节）

**对比**：设计文件将参考文件的理论转化为可自动生成的配置，实现了从方法论到工程实践的跨越。

### 2. 四问题判定流

**参考文件**：提出了四个问题的判定流：
1. 可形式化吗？
2. 代价多高？
3. 反馈要多快？
4. 频率多高？

**设计文件**：在决策引擎（第6.1节）中完整实现了四问题判定流：
- `_checkFormalizable()`：检查可形式化性
- `_adjustCost()`：调整代价（考虑团队规模）
- `_estimateFrequency()`：估计频率（考虑项目阶段和团队规模）
- `_finalDecision()`：综合决策

**对比**：设计文件不仅实现了判定流，还增加了参数调整（团队规模、项目阶段），使决策更加精细化。

### 3. 错误信息设计

**参考文件**：强调"错误信息本身就是 prompt"，应包含三要素：
- 为什么不能这么做
- 应该用什么替代
- 去哪里看正确的例子

**设计文件**：在三段式错误信息生成（第8.2节）中实现了类似结构：
- 原因（why）
- 改为（whatInstead）
- 参考（reference）

**对比**：设计文件已经实现了参考文件的原则，但可能没有充分强调这一原则的重要性。

## 设计文件的优越之处

### 1. 完整的自动化系统

**参考文件**：提供方法论和手动配置指南。

**设计文件**：构建了完整的自动化系统，包括：
- Skill + MCP Server 双层架构
- 8个MCP工具接口
- 状态管理和断点续做
- 备份和回滚机制

**优越性**：将手动过程转化为全自动流程，大大降低了使用门槛。

### 2. 代码扫描与规则发现

**参考文件**：主要依赖手动添加规则。

**设计文件**：提供 `scan_codebase` 工具（第3.1.2节），可以：
- 扫描代码库发现潜在规则
- 提取现有规则
- 提供规则建议

**优越性**：自动化规则发现，减少手动配置工作量。

### 3. 冲突检测与解决

**参考文件**：未提及规则冲突检测。

**设计文件**：提供冲突检测器（第6.2节）：
- 检测直接冲突、冗余规则、需要细化的规则
- 提供解决建议

**优越性**：避免规则冲突导致的不可预测行为。

### 4. 状态管理与断点续做

**参考文件**：未提及状态管理。

**设计文件**：完整的状态管理系统（第4节）：
- `.harness/state.json` 状态文件
- 状态机：null → evaluated → confirmed → generated → validated
- 断点续做支持

**优越性**：支持用户中途退出后继续，改善用户体验。

### 5. 详细的工程实现

**设计文件**提供了完整的工程化方案：
- 实现路径（MVP分三个阶段）
- 测试场景和验证指标
- 错误码定义和错误处理
- 依赖管理和检查

**优越性**：为实际开发提供了清晰的路线图和质量保证。

## 设计文件的差距

### 1. 认知层 Skills 集成不足

**参考文件**：明确区分约束层和认知层，提出 Skills 作为认知层工具：
- Skills 在 description 字段匹配时按需触发
- 适合领域知识和特定工作流
- 与 CLAUDE.md 的"每次加载"形成对比

**设计文件**：Skill 层主要作为工作流引导，与参考文件中的 Skills 概念不同：
- 设计文件的 Skill 是流程控制器
- 未充分利用 Skills 的认知层能力

**差距**：对认知层的自动化支持不足。

### 2. 缺少 A/B 测试机制

**参考文件**：强调 A/B 测试 harness 改动的重要性：
- 加规则前后跑同一任务对比
- 防止 harness 膨胀
- 确保每条规则都有价值

**设计文件**：未提及 A/B 测试机制。

**差距**：缺少验证规则有效性的系统化方法。

### 3. 缺少适用性判断

**参考文件**：讨论什么时候不要做 harness：
- 原型阶段：核心抽象未固定
- 一次性脚本：生命周期短
- 维护成本超过收益：规则膨胀

**设计文件**：未涉及这些判断。

**差距**：可能在不适合的场景应用 harness。

### 4. 对错误信息原则强调不足

虽然设计文件实现了错误信息生成，但没有像参考文件那样强调：
- 错误信息是给 Agent 看的上下文工程
- 所有从 harness 流回模型的信号都是引导
- 改写错误信息的成本低但收益高

**差距**：可能未充分优化错误信息的设计。

## 建议的改进方向

### 1. 集成认知层 Skills

**建议**：在自动化系统中集成认知层 Skills：
- 在 `scan_codebase` 中识别需要认知层支持的场景
- 自动生成或建议 Skills
- 在决策引擎中考虑认知层需求

### 2. 添加 A/B 测试框架

**建议**：在系统中集成 A/B 测试：
- 记录规则应用前后的任务表现
- 提供对比分析工具
- 支持规则效果评估

### 3. 增加适用性判断

**建议**：在流程开始时评估项目是否适合应用 harness：
- 检查项目阶段（git 历史、文件结构）
- 评估预期生命周期
- 提供建议而非强制应用

### 4. 强化错误信息优化

**建议**：加强错误信息设计：
- 提供错误信息模板库
- 支持错误信息的 A/B 测试
- 集成错误信息优化建议

## 结论

设计文件 `harness-automation-design.md` (v3.0) 在将 Harness Engineering 方法论工程化方面取得了显著进展，提供了完整的自动化系统、精细化的决策引擎和全面的工程实现。相对于参考文件，它在自动化程度、工程完整性和用户体验方面具有明显优势。

v3.0 版本已经解决了之前版本的主要差距：
1. **认知层 Skills 集成**：新增第14节"认知层Skills集成"，定义了诊断型、教育型、决策支持型三种认知层Skill类型
2. **A/B 测试框架**：新增第13节"A/B测试与效果评估"，提供了完整的测试配置、数据收集、统计分析和推荐系统
3. **适用性判断**：新增 `assess_suitability` 工具（第3.1.9节），可识别原型阶段、一次性脚本等不适合应用harness的场景
4. **错误信息优化**：扩展第8.2节为"错误信息优化框架"，包含模板库系统、A/B测试集成和优化建议工具

这些改进使系统更加完善，实现了harness的完整生命周期管理，从适用性评估到效果验证的闭环。
