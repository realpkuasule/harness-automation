# 为什么我要做 Harness

## 两种速度

我以前的生活由三件事组成：开会、出差、喝酒。

那不是一种能建造东西的生活。你能谈下合作、摆平分歧、在正确的时间把正确的人拉到一个桌上——但你始终在"促成"，不在"建造"。促成是靠别人的手，建造是靠自己的手。这是两种完全不同的存在方式。

然后 AI 编程工具出现了。我用中文写了一段描述，代码出现在屏幕上，跑起来了。

那个瞬间，我体验的是一种范畴的转换：**一个只会促成的人，突然也可以建造了。**

但建造之后的问题，比建造之前更深。

## 一百亿 token 之后

三个月，一百亿 token 以上。

Claude Opus、GLM、DeepSeek——哪个模型强我用哪个。不是"试一下"，是当成了我的主要工程生产力。每天十几个小时泡在里面，模型一版一版地换，代码一批一批地出。

一百亿 token 大概是什么概念？如果按 token 到有效代码行来折算，这大致相当于一个小团队好几年的产出量。但重点不是量。重点是，在这个量级上，你会看到一些在小样本里根本不会浮现的模式。

第一个模式：**产出越高，控制力越弱。**

每天几百上千行代码往外冒。功能堆叠功能，模块互相咬合。但从某一刻起，我已经不真正理解自己的代码库了。不是看不懂语法——是看不清边界。哪些地方有防御，哪些地方是裸奔，哪些写法是故意的，哪些纯粹是模型顺手那么一写。

第二个模式：**换模型解决不了这个问题。**

Opus 写得好一点，GLM 推理强一点，DeepSeek 速度快一点。但无论是谁，只要你不给它边界，它就在发散的轨道上勤勉地狂奔。好模型的发散只是更工整的发散——注释写得更好、函数拆得更合理——但它依然在发散。问题不出在模型能力上，出在"人与 AI 的协作结构"上。

这些模式让我开始琢磨一个更底层的问题。

人类历史上有过几次类似的速度飞跃：

- 蒸汽机让制造速度提升了百倍，然后就有了标准化。你不能让每个工厂自己决定螺丝的尺寸。
- 集装箱让物流速度提升了百倍，然后就有了全球统一的装卸规程。你不能靠码头工人的经验来管理一艘万吨轮。
- Git 让协作速度提升了百倍，然后就有了 CI/CD。你不能靠"合并前互相说一声"来管理几百个分支。

每一次生产力的飞跃，都必然会倒逼出一套新的秩序体系。不是因为控制欲，是因为旧秩序在设计上就不可能覆盖新速度。

现在，AI 让代码产出速度提升了十倍以上。而质量控制还是靠什么？Code Review。

但我观察到一个比"来不及"更严重的问题：**不是来不及做，是越来越多的人已经不做了。**

焦虑在改变所有人的行为模式。当每个人都觉得"我必须更快、我必须产出更多"，Code Review 就从"必须的工序"变成了"可以牺牲的仪式"。没有人公开反对 Code Review——大家依然点头说它重要——但事实上，它正在悄悄地从日常行为中消失。原先安全的铁则，在速度的洪流里，被无声地抛在脑后。

这不是加人的问题。这是旧范式已经在执行层面崩解了，只不过还没人正式宣布它的死亡。

## 编程的本质是收敛

我停下来想了想一个更基本的问题：编程这件事本身，到底是什么？

从数学形象上看，编程是一个**收敛过程**。

需求的生发是发散的——用户什么都想要，什么都觉得重要。需求的确认是收敛——哪些是核心场景，哪些是边缘情况，哪些留到下一版。

技术调研是发散的——查文档、看方案、读源码。技术选型是收敛——选一个方向，放弃其他。

设计是发散的，方案评审是收敛的。
计划是发散的，任务编排是收敛的。
编码是发散的，测试是收敛的。测试结果出来，有 bug，编码再发散，修完再收敛。

一轮又一轮，直到系统收敛到一个满足需求的状态。

**一个好的编程过程，是让收敛高效发生。一个坏的编程过程，是让发散持续失控。**

回到 AI Coding。这里有一个根本性的不对称：

**AI 是一个发散引擎。** 你给它一句话，它还你五百行。你再给它一句话，它又还你五百行。它不是懒惰的，它极其勤勉——它是无限勤奋的发散。

但人，天然应该是那个收敛引擎。判断什么该要、什么该砍、哪里需要约束、哪里可以自由。

问题在于——当 AI 的发散速度是人的十倍、百倍，人的收敛能力如果不借力于系统，就被彻底淹没。你来不及判断，来不及收束，来不及画边界。你以为你在指挥，其实你只是在追赶。

所以 AI Coding 时代真正的危险不是"代码质量差"，而是**收敛跟不上发散**。发散比收敛快一个数量级，发散就永远是领先的。你永远在追，永远追不上。

这个洞察一出来，很多事就清楚了。

什么是有用的约束？**约束就是收敛的自动化。** 一条"禁止 console.log"的 lint 规则，本质上是把"判断这里该不该打 log"这个收敛动作交给了机器。一条"所有 API 必须返回统一数据结构"的规范，本质上是把"这个返回值和那个返回值格式不一样"的发散可能性预先锁定。

把足够多的、边界清晰的收敛动作交给系统，人的收敛能力才能聚焦到机器做不了的事上——架构的取舍、设计的品味、需求的优先级。

## 河与岸

这让我重新理解了"约束"这个词。

一个常见的回答是：约束是限制。让你不能干这个，不能干那个。

但如果你仔细想，这不是真的。河岸不限制河——河岸定义了河。没有岸的河，是洪水。洪水没有方向，没有深度，哪里都去，哪里都到不了。河有了岸，才有流向，才有流量，才有力量。

约束不是禁止，是指向。

这个区分非常重要，因为它决定了你构建约束体系时的所有选择：

- 如果约束是"禁止"，你的目标就是堵。堵 console.log、堵裸 fetch、堵硬编码密钥。堵完一个再堵下一个，无穷无尽。
- 如果约束是"指向"，你的目标是定义一个边界清晰的行动空间。在这个空间里，速度可以最大化。碰到空间边缘，系统给出信号、给出理由、给出替代方案。你接受的是教育，不是惩罚。

前者是镣铐。后者是铠甲。

我做了二十年业务，见过太多人用"管"的名义去"困"。也见过顶尖的组织怎么做：不是用流程来替代人的判断，而是用流程来保护人的判断力——让人把脑子花在只有人能做的事情上。

## 一把尺子

所以问题变成了：**给定一条规则，怎么判断它应该放在约束体系的哪个位置？**

我提炼了四个维度。

**第一，可形式化吗？**

有些规则，机器能精确判断。比如"后端所有 API 必须具有同样的数据结构"——拦截器的一次配置校验就能覆盖。再如"禁止 console.log"——正则即可匹配。

有些规则，机器不能判断。比如"优先使用提前返回来减少嵌套"——机器看不出三层的 if-else 和提前返回之间的区别，它只能看到 AST。这需要人。

这不是"能 vs 不能"的二分，而是一个谱系。超过工具能力的规则，必须走另一条路径——从"拦截"转为"教育"。你不能硬拦一个机器判断不了的东西，但你可以反复教。

**第二，代价多大？**

每条规则都是成本。配置成本、维护成本、被拦截时打断心流的成本。你不能什么规则都上。一个好的约束体系不是"越严格越好"，而是在给定的资源和阶段下，做最值得的投入。

原型期，探索速度比一切重要，约束应该轻——几条关键的 linter 规则就够了。
成长期，技术债开始产生利息，该硬化的地方要硬化——hook 和 CI 开始介入。
成熟期，多人多手，混乱的成本急剧上升——约束体系需要完整。

**第三，反馈应该多快？**

有些事，在你写代码的那一刻就必须被拦住——"你在把这个 API key 硬编码进前端"。这个拦截必须在你按下保存键之前发生。晚了就没意义。

有些事，在提交时检查就够了——"你的 commit message 符合规范吗？"

有些事，在合并到主分支时检查也不晚——"测试都通过了没有？"

**第四，这件事多常发生？**

一天触发 50 次，自动化收益巨大，值得你把规则配到尽善尽美。一年触发 2 次，写清楚文档就够了。自动化是成本，不是信仰。

这四把尺子合在一起，构成了 Harness 的决策引擎。每条规则经过这四问，自动落在对应的约束介质上：lint error、lint warn、git hook、CI，或者 CLAUDE.md。

它的价值不在于复杂——刚好相反——**它的价值在于，把原本靠"老王知道""上次踩过坑""别的项目这么配的"来决定的模糊判断，变成了一套可复现、可解释、可演化的决策逻辑。**

## 两层护甲

还有一个维度花了更长时间。

大多数规则工具只做一件事：在代码生成**之后**检查。Lint 在你保存文件后报错，CI 在你 push 之后跑测试。这很好，但有一个根本局限：**它只能告诉你哪里错了，不能帮你不犯错。**

AI Coding 时代有一个以前不存在的机会：你可以在代码被生成**之前**就施加约束。

AI 写代码之前会读上下文。如果你在 CLAUDE.md 里写清楚"所有 API 必须使用 camelCase 字段名，禁止 snake_case"、"字体优先选用本地字体，不可使用中国大陆境外源"、"前后端冲突时，如果后端满足设计需求，不要改后端，总改前端"，AI 在生成代码的那一刻就会遵循这些原则。它不是在写完被 lint 拦下再改——它是从一开始就没那么写。

这是两层护甲，一前一后：

- **前方护甲**：在 AI 读取的上下文里写入规则，影响生成质量。这是一条认知层的防线。
- **后方护甲**：在代码落地的那一刻，ESLint + Hook + CI 三重检查。这是一条执行层的防线。

同一个规则，前后夹击。比任何一个单向工具都有效得多——不只是双保险，是两条独立防线。一条在认知层让 AI 知道什么是好的，一条在执行层让系统保证什么是合格的。

而两层的分工，正是那四把尺子的结果：可形式化的走执行层，不可形式化的走认知层。高频的走近实时拦截，低频的走近延迟检查。

## 再往前走一步：收敛的节奏

有了两层的分工，你就可以进一步思考**收敛的节奏**。

编程不是一次性收敛到位的。它是多轮循环。一个好的 Harness 体系不是在最后一道关才起作用——它在每一轮循环的关节点上都起作用。

你有了一个想法，AI 说"我来帮你 Plan"。Plan 的过程本身就是第一轮收敛。Harness 在这个阶段告诉你：技术栈是什么、约束有哪些、设计规范是什么。AI 读懂了这些，Plan 才不会跑偏。

Plan 完了，分解成任务。任务编排是第二轮收敛。Harness 在这个阶段告诉你：每条任务必须怎么测试——用户测试端到端为主，单元测试和集成测试尽量 AI 自动完成。

任务开始实施，编码阶段是第三轮收敛。Harness 在这里有两层护甲同时作用：AI 在写代码前读了 CLAUDE.md 里的规范，写完的瞬间 linter 和 hook 开始检查。

测试阶段是第四轮收敛。测试结果出来，暴露 gap，再做一轮 gap analysis，再重新计划、重新编排、再实施。每一轮循环都在进一步逼近"满足需求、满足设计"的状态。

**Harness 的真正价值，不是某一个节点的拦截，而是让每一轮收敛都更高效、更少回路。**

这就引出了最后一条原则：**解耦协同。** 不同模块要能独立开发、互不影响。因为只有解耦了，收敛才能局部发生。你修 A 模块不需要等 B 模块通过测试——各自收敛各自的，最后拼在一起。如果收敛必须全局同步，效率就会跌回人肉协作的水平。

## 这个时代真正缺的是什么

AI Coding 解决了"怎么写"的问题。你不需要苦学语法、不需要背 API、不需要记住配置文件的格式。

但有一个问题它永远解决不了。

**"什么算好？"**

这不是技术问题。这是价值观问题。价值观的建立，只能靠人。

在以前，工程师通过一万个小时的刻意练习、踩坑、被骂、读代码、重构、再踩坑，来慢慢建立自己的品味。这是一种缓慢的传递——师徒制、Code Review、技术分享、前人留下的代码风格。

AI 把这个过程的效率基础摧毁了。新人不需要练一万个小时就能写出能跑的代码。一万个小时被压缩成了十个小时。但品味没有压缩——品味不服从速度。你不能加速一个人的判断力形成。

那怎么办？

只有一条路：**把品味编码成结构。把默会知识变成显性约束。把"老王觉得"变成"系统执行"。**

这不是用机器替代人的判断。这是让系统处理"对与错"——那些边界清晰、可精确描述的东西。让人的判断力聚焦在"好与更好"——那些需要品味、上下文、整体感的东西。

你越信任系统做第一层，你的注意力就越完整地留给第二层。**自动化不是替代人，是解放人。收敛不是限制创造，是为创造提供边界。**

## 我为什么做这件事

我用了二十年，在会议室和酒桌上辗转。我推动过很多事情，但从未亲手建造过任何东西。

AI 让我重新拿回了建造的能力。这是我这一代人最大的幸运。

但建造本身不是终点。建造什么、怎么建造、建造的标准是什么——这些问题，工具回答不了，只有人能回答。

三个月前，我已经在有意识地做这件事了——给 AI 协作建立边界，把发散的产出装进收敛的框架里。只是那时候没给它取名字。后来发现这件事可以更系统、可以更自动化、可以让更多像我一样的人不用从头摸索，才有了现在这个工具。

它不是我的"发明"。它是一种实践模式的**识别和系统化**。而这个名字——Harness——是在做的过程中自己浮现出来的。

它不是 linter，不是 CI 工具，不只是把规则生成成配置文件。它是一套**收敛体系**——让你和 AI 协作时，发散有边界，收敛有节奏，每一轮都在向前逼近，而不是原地打转。

它不是笼头。笼头让马停下来。Harness 是马具。戴上它，你才能骑上马，放开腿跑。跑得越快，它越重要。
